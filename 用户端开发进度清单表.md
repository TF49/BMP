# 用户端开发进度清单表

## 文档说明

- 分析范围：`BMP-uniapp` 用户端（含用户可见页面、共享 API、与会长端联动链路）
- 分析维度：
  - 页面是否缺失
  - 是否对接后端真实数据
  - 用户端与会长端是否能联动
  - 数据传输是否存在明显风险
- 状态定义：
  - `已完成`：页面、接口、基础联动基本闭环
  - `部分完成`：已有页面和接口，但仍有明显缺口
  - `仅UI/假数据`：主要为静态展示或模拟流程
  - `异常`：存在明显错配、死链、配置问题或高风险缺陷

---

## 一、总体结论

| 维度 | 当前判断 | 结论 |
| --- | --- | --- |
| 页面铺设 | 较高 | 用户端主要页面文件基本齐全，但存在死链接和双份页面配置分叉 |
| 真实数据对接 | 中等 | 列表、详情类页面已接入较多真实接口，但首页、帮助、确认页、赛事列表仍有静态/模拟逻辑 |
| 用户端-会长端联动 | 中等偏低 | 双方大量共用同一后端资源，但会员身份映射、支付确认、充值链路不稳定 |
| 数据传输稳定性 | 偏低 | 存在 `userId/memberId` 混用、参数名不一致、接口不存在、类型未收敛等问题 |

### 总体评级

| 模块 | 进度判断 |
| --- | --- |
| 页面建设 | 80%+ |
| 后端真实数据接入 | 50%~60% |
| 用户端与会长端联动 | 40%~50% |
| 数据传输与接口契约稳定度 | 35%~45% |

## 二、全局问题清单

### 1. 路由与页面配置问题

| 问题 | 影响 | 结论 |
| --- | --- | --- |
| `BMP-uniapp/pages.json` 与 `BMP-uniapp/src/pages.json` 内容不一致 | 可能导致不同构建入口下页面注册结果不同 | 高风险 |
| `BMP-uniapp/src/pages.json` 含尾逗号，JSON 非法 | 影响配置解析、构建稳定性 | 异常 |
| 首页存在跳转 `/pages/insights/detail`，但项目中无该页面 | 用户点击后会跳转失败 | 异常 |

### 2. 会员身份映射问题

| 问题 | 影响 | 结论 |
| --- | --- | --- |
| 多个用户端页面直接把 `userId` 当 `memberId` 使用 | 容易查错会员信息、创建错单、导致用户端与会长端数据错位 | 高风险 |
| 后端已提供 `/api/member/current`，前端基本未采用 | 当前登录用户与会员实体未做稳定绑定 | 高风险 |

### 3. 交易闭环问题

| 问题 | 影响 | 结论 |
| --- | --- | --- |
| 场地预约确认页是前端模拟完成，不是真正支付/状态流转 | 用户端显示成功，但后台订单状态未必同步 | 高风险 |
| 充值页前端参数 `paymentMethod` 与后端字段 `method` 不一致 | 用户自助充值大概率失败 | 高风险 |
| 帮助反馈页是前端 `setTimeout` 假提交 | 用户反馈无法真实入库 | 中高风险 |

### 4. 接口契约问题

| 问题 | 影响 | 结论 |
| --- | --- | --- |
| 搜索建议接口前端已调用，后端未实现 `/search/suggestions` | 搜索建议无法正常工作 | 高风险 |
| 类型检查错误数量较多 | 表示 API 字段、页面字段、类型定义尚未统一 | 高风险 |
| 部分页面存在 mock 回退逻辑 | 容易误判开发进度 | 中风险 |

## 四、用户端与会长端联动分析

### 1. 已具备联动基础的模块

| 模块 | 用户端接口 | 会长端接口 | 是否共用后端资源 | 结论 |
| --- | --- | --- | --- | --- |
| 场馆 | `/venue/list` `/venue/info` | 会长端场馆管理接口 | 是 | 可联动 |
| 场地预约 | `/booking/*` | 会长端 `/booking/*` | 是 | 可联动，但支付确认链路不完整 |
| 课程 | `/course/*` `/course/booking/*` | 会长端课程与课程预约管理 | 是 | 可联动 |
| 器材租借 | `/equipment/*` `/equipment/rental/*` | 会长端器材与租借管理 | 是 | 可联动 |
| 赛事 | `/tournament/*` `/tournament/registration/*` | 会长端赛事与报名管理 | 是 | 可联动 |
| 穿线 | `/stringing/*` | 会长端穿线管理 | 是 | 可联动 |
| 通知 | `/notifications` | 会长端通知管理 | 是 | 可联动 |

### 2. 联动未闭环的关键点

| 问题 | 说明 | 影响 |
| --- | --- | --- |
| 用户端订单确认页是假流程 | 用户操作完成后，后台状态不一定同步变化 | 用户端与会长端看到的订单状态可能不一致 |
| `userId` 直接当 `memberId` 使用 | 用户实体与会员实体未稳定映射 | 会长端查到的会员订单可能不是当前用户真实会员记录 |
| 充值字段不一致 | 用户端充值可能提交失败或后端识别不到支付方式 | 会长端财务侧无法收到真实新增充值 |
| 搜索建议、反馈等能力未接通 | 用户端表面可操作，后台无真实记录 | 管理端无法接收对应数据 |

## 五、数据传输风险清单

| 风险类型 | 具体问题 | 风险级别 |
| --- | --- | --- |
| 参数名错配 | `paymentMethod` vs `method` | 高 |
| 身份错配 | `userId` 当 `memberId` 使用 | 高 |
| 接口缺失 | `/search/suggestions` 前端调用但后端不存在 | 高 |
| 配置错误 | `src/pages.json` 非法 JSON | 高 |

| Mock 数据回退 | 赛事列表失败时回退演示数据 | 中 |
| 类型未收口 | `type-check-errors.txt` 存在大量错误 | 高 |

## 六、优先修复建议

### P0：必须先修

1. 统一并修复 `pages.json` / `src/pages.json`
2. 建立“当前登录用户 -> 当前会员”的统一解析逻辑，优先改用 `/api/member/current`
3. 修复充值参数契约，统一 `method/paymentMethod`
4. 取消预约确认页假完成流程，改为真实支付/状态更新流程

### P1：应尽快修

1. 移除或显式标记首页、赛事页中的静态演示数据
2. 补齐 `/search/suggestions` 或去掉前端调用
3. 将帮助页反馈改为真实提交 `/api/feedback`
4. 修复所有 `userId/memberId` 直接混用页面

### P2：后续收尾

1. 收敛 TypeScript 类型错误
2. 清理“功能开发中”占位逻辑
3. 给核心下单、充值、报名、租借流程补回归测试

## 七、最终结论

当前用户端不是“没做完页面”，而是“页面做得比业务闭环快”。  
也就是说：

- **UI 页面层完成度已经不低**
- **真实数据对接完成了一半左右**
- **用户端与会长端确实共用了大量后端资源**
- **但会员身份、支付确认、充值、搜索建议、帮助反馈这些关键链路仍未收口**

因此现阶段更准确的判断是：

> 用户端已进入“联调修正阶段”，而不是“继续堆页面阶段”。

后续最应该投入的是接口契约统一、真实业务闭环和联动回归，而不是继续新增视觉页面，老公。
