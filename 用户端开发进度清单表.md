# 用户端开发进度清单表

## 文档说明

- 分析时间：2026-04-22
- 分析范围：`BMP-uniapp` 用户端主链路，以及与会长端共用后端资源的联动情况
- 本次结论依据：
  - `BMP-uniapp/pages/**`
  - `BMP-uniapp/api/**`
  - `BMP-uniapp/composables/useCurrentMember.ts`
  - `BMP-uniapp/store/modules/user.ts`
  - 后端 `src/main/java/**` 对应控制器
- 状态定义：
  - `已完成`：页面、接口、主流程闭环基本成立
  - `部分完成`：页面与接口已存在，但仍有联调缺口或稳定性问题
  - `仅UI/静态`：以展示为主，未形成真实业务链路
  - `异常`：存在明显契约错误、错误路由、错用身份或高风险行为

| 维度 | 当前判断 | 结论 |
| --- | --- | --- |
| 页面铺设 | 高 | 用户端主页面基本齐全，`pages.json` 与 `src/pages.json` 已一致，页面注册层面明显好于上一版判断 |
| 真实数据对接 | 中高 | 首页余额/预约提醒、会员中心、预约、课程、器材、充值记录、搜索结果等已接入较多真实数据 |

| 数据传输稳定性 | 中等偏上 | 关键链路比上一版更完整，搜索建议、帮助反馈、赛事真实空态、用户侧穿线取消已接通，但支付退款回归与类型定义仍未完全收口 |

### 阶段判断

> 当前用户端已经不是“页面建设阶段”，而是比较明确地进入了“联调收口阶段”。

与上一版文档相比，变化最大的地方有三点：

1. `pages.json` / `src/pages.json` 已经同步，不再存在非法 JSON 问题
2. `useCurrentMember` 已经在多数主链路页面落地，`userId` 与 `memberId` 混用不再是全局性问题
3. 场地预约确认页已接真实支付接口，不再是纯前端假完成页

- 页面层基本已经铺满
- 核心真实数据接入已经明显超过“半成品”阶段

> 用户端当前开发进度整体约为 84% 左右，主任务已经从“继续补页面”切换为“补联调、做回归、压类型问题”。
