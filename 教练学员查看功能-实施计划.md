# 教练学员查看功能 · 实施计划

## P0 冻结覆盖说明

本节为最终冻结决策，优先级高于本文后续早期草案中的冲突描述：

- 首版交付 P1—P5，首版 P1—P5 不实现 WebSocket；P6 为独立增强，不阻塞上线。
- 教练—学员关系由有效课程预约推导，历史权限保留至最后一次有效课程结束后 180 天。
- 列表只返回 `maskedPhone`；详情完成关系鉴权后才返回完整 `phone`；身份证只返回 `idCardTail`。
- 不返回余额、充值额、充值记录或其他课程消费；只返回可关联到当前教练课程预约的 COURSE 净消费。
- 考勤独立于预约状态：0未登记、1已签到、2已完成、3缺席；不得自动把未签到推断为缺席。
- 旧 `status=4` 只迁移为 `attendance_status=2`，取消记录不得迁移为缺席。
- `/for-coach/status` 只允许课程开始前取消；签到、完成、缺席统一走考勤 action 命令接口。
- 自动结束时考勤1转2、考勤3保持3、考勤0保持0。
- 越权异常统一返回403，Controller 不得捕获 AccessDeniedException 并转换为普通业务响应。
- 重查询超时：有最后成功缓存时返回旧缓存；无缓存时返回503和 traceId，禁止伪造空分页。
- 头像统一复用 `/static/placeholders/avatar.svg`；教练底栏保持4项，工作台增加全宽“我的学员”入口。

## 0. 全局设计决策（P0 阶段 · 评审通过后方可启动 P1）

| 决策项 | 选择 | 理由 / 备注 |
|--------|------|-------------|
| 教练 ↔ 学员关联 | **复用 `biz_course → biz_course_booking` 链路**，不新增硬关联表 | 现项目"教练带学员"的所有数据天然存在于课程预约记录；如后续支持场地预约教练代课，再补 `biz_booking_coach_rel`。 |
| 课程包 vs 单次课 | **先做"单次课 + 按课程名称聚合进度"的轻量方案（方案 A）**，为未来课程包留 DDL 占位 | 当前 `biz_course` 是"一节课 = 一个 courseId"粒度，直接做课程包会牵涉订单/预约/扣课时的全链路改造，风险过大。 |
| 学员头像来源 | **通过 `sys_member.user_id → sys_user.avatar` 关联**，`sys_member` 不扩展 avatar 字段 | 与项目现有教练头像展示方式一致；返回时用 `CASE WHEN u.avatar IS NOT NULL THEN u.avatar ELSE '默认常量' END`。 |
| 接口命名空间 | **新增 `/api/coach/students/*` 独立一组（COACH 专属）**，不共用 `/api/member/*` | 避免通用 member 接口权限粒度过粗导致越权；也便于以后做教练专属字段扩展。 |
| 越权策略 | **白名单制 + 缓存加速**：必须在 `biz_course_booking` 中存在当前教练的交集记录，否则 403；越权校验结果缓存5分钟 | 兼容"会员属于其他场馆、教练跨场馆代课"两种情况，比按 `venueId` 粗暴过滤更精确；缓存避免高频查询成为性能瓶颈。 |
| 现有 `students.vue` 语义 | **保留为“单课程签到/预约学员名单”页面**，不承载“我的学员”全局列表；仅在每条学员行增加「查看学员详情 →」跳新详情页 | 避免破坏已经跑通的签到链路；新全局列表使用 `student-list.vue`，详情使用 `student-detail.vue`，页面语义不冲突。 |
| 实时更新 | **onShow + refresher 兜底**，P6 再叠加 WebSocket；首版 P1~P5 不依赖 WebSocket 也能跑通 | 与现有 `courses.vue` 模式保持一致；出了线上问题能快速回退。 |
| 敏感数据策略 | 手机号**列表页脱敏展示、详情页"拨打确认弹窗"显示全号**，后端返回全量；身份证号只返末 4 位 | 平衡隐私保护与业务需求：列表页防止泄露、拨打前二次确认增强安全感；身份证对教练是低需求字段，优先最小化暴露。 |
| 工作台入口布局 | **在现有3个横向卡片下方新增全宽"我的学员"卡片**，不改现有布局为2×2网格 | 现有工作台已采用 `grid-template-columns: repeat(3, minmax(0, 1fr))` 的3列布局，改为2×2会破坏现有视觉平衡；新增全宽卡片风险更小。 |
| 课程预约状态 vs 考勤状态 | **保留现有 `biz_course_booking.status`：0已取消、1待支付、2已支付、3进行中、4已完成；新增轻量 `attendance_status` 表示考勤：0未签到、1已签到/进行中、2已完成、3缺席** | 当前项目没有“缺席”预约状态，禁止把 `status=4` 当缺席；缺席必须通过独立考勤字段表达，否则出勤率无法可靠统计。 |
| 出勤率计算规则 | **出勤率 = 出勤完成数 / (出勤完成数 + 缺席数)**；完成数取 `attendance_status=2`，缺席数取 `attendance_status=3`；历史无考勤字段的数据仅可把 `status=4` 回填为完成，不自动推断缺席 | 统一前后端口径，避免误用现有预约状态码；明确排除待支付/已支付/进行中/已取消等非考勤完成态。 |
| 风险学员阈值 | **出勤率 < 70% 或 最近14天未上课**，支持配置化（`application.yml`）；“最近上课”按 `biz_course.course_date + end_time` 中最近一次 `attendance_status=2` 或历史 `status=4` 计算 | 首版使用固定阈值快速上线；不能用 `cb.update_time` 代表上课时间，避免状态更新时间污染风险判断。 |

### 1.4 DDL 增量建议（写入 `patch-coach-students.sql`）

```sql
-- ============================================================
-- 教练视角学员查询 · 新增考勤字段 + 索引
-- 依赖：badminton.sql 现有的 biz_course / biz_course_booking / sys_member / sys_user
-- 版本：v1.2（修正预约状态码、MySQL兼容性、上课时间口径）
-- ============================================================

-- MySQL 兼容性要求：
-- 1. 项目当前 SQL 为 MySQL 风格，不能直接使用 CREATE INDEX IF NOT EXISTS。
-- 2. 正式脚本必须先查 information_schema.columns / information_schema.statistics，
--    确认字段或索引不存在后再执行 ALTER TABLE / CREATE INDEX。
-- 3. 下方为目标 DDL 片段，不可原样无判断地重复执行。

-- 关键：当前 biz_course_booking.status 没有“缺席”含义；
-- 保留预约状态码：0已取消、1待支付、2已支付、3进行中、4已完成；
-- 新增独立考勤字段，避免把 status=4 误判为缺席。
ALTER TABLE biz_course_booking
    ADD COLUMN attendance_status TINYINT(1) NULL DEFAULT 0 COMMENT '考勤状态（0-未签到，1-已签到/进行中，2-已完成，3-缺席）' AFTER status,
    ADD COLUMN actual_checkin_time DATETIME NULL COMMENT '实际签到时间' AFTER attendance_status,
    ADD COLUMN actual_finish_time DATETIME NULL COMMENT '实际完成时间' AFTER actual_checkin_time;

-- 关键：教练想"看学员"需要通过 course → course_booking 反查
CREATE INDEX idx_cb_course_member_status
    ON biz_course_booking (course_id, member_id, status, attendance_status, del_flag);

-- 出勤率计算专用索引（按学员+考勤状态查询）
CREATE INDEX idx_cb_member_attendance
    ON biz_course_booking (member_id, attendance_status, status, del_flag, course_id);

-- 教练课程时间范围查询优化索引；注意字段名为 start_time/end_time，不是 course_start_time/course_end_time
CREATE INDEX idx_c_coach_date_range
    ON biz_course (coach_id, course_date, start_time, end_time, status, del_flag);

-- 教练课程基础索引；badminton.sql 已存在 idx_coach_date(coach_id, course_date)，
-- 若 DBA 评估 idx_c_coach_date_range 覆盖充分，可不重复创建此索引。
CREATE INDEX idx_c_coach_date
    ON biz_course (coach_id, course_date, start_time, del_flag);

-- 列表搜索：支持教练按姓名/手机号模糊搜其学员
CREATE INDEX idx_sm_name_include
    ON sys_member (member_name, del_flag);
CREATE INDEX idx_sm_phone_include
    ON sys_member (phone, del_flag);

-- 风险学员筛选按“课程日期+结束时间”判断，不使用 cb.update_time 代表上课时间。
CREATE INDEX idx_cb_course_attendance
    ON biz_course_booking (course_id, attendance_status, status, del_flag);

-- 越权校验加速索引（高频EXISTS查询优化）
CREATE INDEX idx_c_coach_id_only
    ON biz_course (coach_id, id, del_flag);

-- ============================================================
-- 数据回填脚本（首次部署必须执行，用于历史数据补全）
-- ============================================================

-- 历史数据：现有 status=4 表示“预约已完成”，可回填为 attendance_status=2。
-- 不能把取消记录自动推断为缺席；缺席只能由后续教练明确标记。
UPDATE biz_course_booking
SET attendance_status = 2,
    actual_finish_time = COALESCE(actual_finish_time, update_time)
WHERE del_flag = 0
  AND status = 4
  AND (attendance_status IS NULL OR attendance_status = 0);

-- 不在 sys_member 上回填 first_lesson_time / last_lesson_time：
-- 当前 sys_member 无这两个字段，列表/详情接口应通过聚合查询按需返回。
```

#### 1.4.1 DDL 执行与回滚要求
- **幂等字段判断**：执行 `ALTER TABLE` 前必须查询 `information_schema.columns`，确认
  `attendance_status / actual_checkin_time / actual_finish_time` 不存在；若任一字段已存在，脚本必须跳过对应字段。
- **幂等索引判断**：执行 `CREATE INDEX` 前必须查询 `information_schema.statistics`，确认索引名不存在；不得依赖 `CREATE INDEX IF NOT EXISTS`。
- **线上大表策略**：如 `biz_course_booking` 行数超过 10 万，DBA 需评估在线 DDL 能力；优先使用数据库原生 online DDL 或维护窗口执行，避免锁表影响下单/签到。
- **回填批处理**：历史 `status=4` 回填为 `attendance_status=2` 时按主键范围分批执行，每批建议 1000~5000 行；每批记录影响行数，失败后可从最后成功 id 继续。
- **回滚策略**：字段新增后不立即删除；若上线后需回退应用，保留新增字段和索引不影响旧代码。只有确认无代码依赖后，另发 DBA 审核脚本删除字段/索引。
- **验证 SQL**：
  ```sql
  SELECT status, attendance_status, COUNT(*)
  FROM biz_course_booking
  WHERE del_flag = 0
  GROUP BY status, attendance_status;
  ```
  验收标准：历史 `status=4` 的记录应主要落在 `attendance_status=2`；`status=0` 不得被批量回填为 `attendance_status=3`。

### 1.5 编码硬性约束
- 所有新 SQL 的 coachId 条件，统一使用
  `INNER JOIN biz_course c ON cb.course_id = c.id AND c.coach_id = ?`
  （与现有 `findAllByCoachId` 写法一致），禁止改从 `venue_id` 间接推断。
- `attendanceRate` 计算必须使用 `NULLIF(..., 0)`，除数为 0 时返回 0 或 NULL，避免 SQL 报错。
  **统一算法**：`ROUND(attendedCount * 100.0 / NULLIF(attendedCount + absentCount, 0), 2)`
  - `attendedCount`：`attendance_status = 2` 的记录数；历史兼容时可把 `status = 4` 且 `attendance_status` 为空/0 的记录视为完成。
  - `absentCount`：`attendance_status = 3` 的记录数。
  - **不计入**：`status IN (0, 1, 2, 3)` 且 `attendance_status NOT IN (2, 3)` 的记录；取消、待支付、已支付、进行中都不是出勤率分母。
  - **预约状态码禁止改义**：`biz_course_booking.status=4` 在当前项目中是“已完成”，不是“缺席”。
- 所有新方法在 `CourseBookingMapper` 注释中标明「仅用于教练端」，防止以后在管理员/会员接口里被错误复用导致性能问题。
- **风险学员标记逻辑**（后端统一计算，前端禁止重复实现）：
  ```java
  isRisk = (attendanceRate < riskAttendanceRate) 
           || (lastLessonTime != null && lastLessonTime.isBefore(LocalDate.now().minusDays(riskInactiveDays)))
  ```
  其中 `riskAttendanceRate`、`riskInactiveDays` 从配置读取（见 P0 决策表）。

### 1.6 考勤字段写入链路（必须随 P1/P2 同步落地）
新增字段不是只供查询使用，必须在现有教练签到/完成链路中同步维护：

| 操作入口 | 当前预约状态变化 | 新考勤字段写入 | 实际时间字段 |
|----------|------------------|----------------|--------------|
| 教练点击「签到」 | `status: 2 → 3` | `attendance_status = 1` | `actual_checkin_time = NOW()`，仅当原值为空时写入 |
| 教练点击「完成」 | `status: 3 → 4` | `attendance_status = 2` | `actual_finish_time = NOW()`，仅当原值为空时写入 |
| 教练点击「缺席/取消」且课程已开始或教练明确选择缺席 | `status` 可保持业务允许的取消流转或置 `0` | `attendance_status = 3` | `actual_finish_time` 不写；`cancel_time` 仍按取消逻辑写 |
| 学员/系统取消待上课预约 | `status = 0` | 保持 `attendance_status = 0`，不得自动算缺席 | `cancel_time = NOW()` |

实现要求：
- `CourseBooking` 实体新增：
  ```java
  private Integer attendanceStatus;

  @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
  private LocalDateTime actualCheckinTime;

  @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
  private LocalDateTime actualFinishTime;
  ```
- `CourseBookingMapper#findById / findByIdAndCoachId / findAllByCoachId / findStudentScheduleForCoach`
  必须返回上述 3 个字段，确保前端可区分预约状态与考勤状态。
- `CourseBookingMapper` 新增或改造状态更新方法，不再只更新 `status/remark/update_time`：
  ```java
  int updateCoachBookingStatusAndAttendance(
      @Param("id") Long id,
      @Param("status") Integer status,
      @Param("attendanceStatus") Integer attendanceStatus,
      @Param("remark") String remark,
      @Param("writeCheckinTime") boolean writeCheckinTime,
      @Param("writeFinishTime") boolean writeFinishTime
  );
  ```
- `CourseBookingServiceImpl#updateStatusForCoach` 必须把 `status=3` 映射为 `attendanceStatus=1`，
  `status=4` 映射为 `attendanceStatus=2`；缺席不得复用 `status=4`，需要独立参数或独立接口明确表达。
- 如首版不想新增缺席接口，前端按钮文案应从「缺席/取消」拆成「取消」和「登记缺席」，避免一个按钮触发两种业务含义。

| GET | `/api/coach/students/{id}` | 学员详情（基础档案 + KPI 聚合 + 等级/到期；余额为 P0 审批后的可选字段） | member `findById` + `getStudentCoachRelationStats` | **必须 `isStudentTaughtByCoach(coachId, id)`**，否则 403 |
| GET | `/api/coach/students/{id}/progress` | 训练进度（按课程名分组） | `getStudentProgressByCourseNameForCoach` | 同上越权校验 |
| GET | `/api/coach/students/{id}/schedule` | 课程安排（时间区间 + status + 分页） | `findStudentScheduleForCoach` | 同上越权校验 |
| GET | `/api/coach/students/{id}/attendance` | 出勤记录（时间线 + 月度汇总） | `findStudentAttendanceTimelineForCoach` + 月度聚合 | 同上越权校验 |
| GET | `/api/coach/students/{id}/consume-records` | 仅查看**与当前教练课程相关**的消费记录 | 新增教练范围消费查询，禁止直接透传通用 member 全量消费接口 | 同上越权校验 + 原接口分页 |

> 隐私边界：首版不向教练开放学员充值记录明细；若详情页展示余额/累计消费，必须由产品确认字段范围。`totalRecharge` 默认不返回，除非后续 P0 单独评审通过。

### 2.3.1 参数校验与接口防护（六道防线落地）

| 防线 | 要求 | 失败处理 |
|------|------|----------|
| 参数校验 | `memberId/id/courseId > 0`；`page >= 1`；`1 <= size <= 100`；`keyword` trim 后最长 50；`status IN (0,1,2,3,4)`；`attendanceStatus IN (0,1,2,3)`；`orderBy` 仅允许 `lastLessonTime/attendanceRate/totalPaidAmount/totalHours`；日期区间最大 180 天 | 参数非法返回 400/业务错误码，不进入 Mapper |
| 业务规则 | 当前登录用户必须已绑定教练档案；学员未逻辑删除；课程/预约未逻辑删除；时间区间 `start <= end` | 返回业务错误，不暴露 SQL 或内部异常 |
| 权限校验 | 列表只能使用当前 coachId；所有 `{id}` 端点必须先调 `ensureCoachCanViewStudent` | 无关系返回 403 |
| 幂等处理 | GET 接口天然幂等；登记签到/完成/缺席沿用预约状态机，必须基于旧状态条件更新 | 重复完成/重复缺席返回明确业务提示 |
| 请求限流 | `{id}` 详情类接口按 `coachId + memberId` 限制 10 次/秒；列表接口按 `coachId` 限制 5 次/秒；异常高频 memberId 枚举进入告警 | 超限返回 429 或项目统一错误码 |
| 降级与异常 | `progress/attendance` 缓存失效或慢查询时降级为近 90 天；消费记录接口不可降级为全量 member 查询 | 返回空分页或降级提示，不返回敏感全量数据 |

### 2.4 Service 层要点
1. **越权复用 + 缓存优化**：抽 `canCoachViewStudent(Long coachId, Long memberId)` 做 boolean 缓存；
   再由 `ensureCoachCanViewStudent(Long coachId, Long memberId)` 统一抛 `AccessDeniedException`。
   所有 `{id}` 端点首行调用 `ensure...`，避免 5 个方法各自复制。
   ```java
   // 在 CoachStudentRelationServiceImpl 中定义；避免 CoachServiceImpl 内部自调用导致 @Cacheable 代理不生效。
   @Cacheable(value = "coach:student:relation", key = "#coachId + ':' + #memberId")
   public boolean canCoachViewStudent(Long coachId, Long memberId) {
       if (coachId == null || memberId == null) return false;
       return courseBookingMapper.isStudentTaughtByCoach(coachId, memberId) > 0;
   }

   public void ensureCoachCanViewStudent(Long coachId, Long memberId) {
       if (!canCoachViewStudent(coachId, memberId)) {
           throw new AccessDeniedException("无权查看该学员信息");
       }
   }
   
   // 缓存 false 也有价值，可缓解恶意枚举 memberId 导致的穿透；TTL 统一 5 分钟。
   // 缓存失效时机：学员新增/取消预约、教练离职/解绑、会员注销/逻辑删除。
   @CacheEvict(value = "coach:student:relation", key = "#coachId + ':' + #memberId")
   void evictStudentRelationCache(Long coachId, Long memberId);
   ```
2. **高风险标记统一算**：`isRisk = attendanceRate < 70 OR lastLessonTime 超过 14 天`，
   在 Service 层统一写入返回对象，**禁止前端各自实现**（否则不同页面对同一学员显示结果不一致）。
   ```java
   // 读取配置
   @Value("${coach.risk-threshold.attendance-rate:70}")
   private int riskAttendanceRate;
   
   @Value("${coach.risk-threshold.inactive-days:14}")
   private int riskInactiveDays;
   
   // 统一判断逻辑；attendanceRate 为 null 时按 0 处理，但不要把无历史课的学员误判为低出勤。
   private boolean isRiskStudent(CoachStudentItem student) {
       if (student.getAttendedCount() + student.getAbsentCount() > 0
               && student.getAttendanceRate() < riskAttendanceRate) {
           return true;
       }
       if (student.getLastLessonTime() == null) return false;
       LocalDate lastLesson = student.getLastLessonTime().toLocalDate();
       return lastLesson.isBefore(LocalDate.now().minusDays(riskInactiveDays));
   }
   ```
3. **缓存策略**：对 `progress`、`attendance` 等查询较重的接口，加基于 `coachId + memberId`
   的 `@Cacheable`（参考 `CourseEntityCache` 模式）；并在
   `CourseBookingService#updateStatusForCoach` 成功路径里清对应缓存。
4. **字段裁剪**：
   - 手机号原样返回（教练要拨打电话，前端统一脱敏展示）；
   - 身份证号仅返末 4 位（`idCardTail`），不返整串；
   - `avatar` 为空时由 Service 层替换为前端可识别的占位常量 URL（保持与教练头像占位一致的命名）。
   - 充值记录明细默认不返；`balance/totalRecharge` 属于账户财务信息，必须按 P0 评审结果决定是否返回。若未评审通过，详情页只返回与当前教练课程相关的 `totalPaidAmount/totalConsumptionForCoach`。

export function maskPhone(phone?: string, showFull = false): string {

### 6.1 WebSocket（P6，**不影响 P1~P5 独立上线**）

#### 新增事件常量（建议枚举）
| 事件 key | 触发点 | 携带 payload | 优先级 |
|----------|--------|--------------|--------|
| `COACH_STUDENT_CANCELLED` | 学员取消我的课 | memberId、bookingId、courseName | **HIGH（立即弹窗+震动）** |
| `COACH_STUDENT_NEW_BOOKING` | 新学员报名我的课 | memberId、courseId、courseName | MEDIUM（Badge提示） |
| `COACH_STUDENT_ATTENDANCE_CHANGED` | 签到/完成/缺席变化 | memberId、bookingId、bookingStatus、attendanceStatus | LOW（仅刷新列表） |
| `COACH_MEMBER_STATUS_CHANGED`（可选） | 会员到期/冻结 | memberId、新状态 | LOW |

#### 前端订阅处理（分级响应策略）
```ts
// 在 store/modules/coach-ws.ts 中定义
const WS_EVENT_PRIORITY = {
  COACH_STUDENT_CANCELLED: 'HIGH',
  COACH_STUDENT_NEW_BOOKING: 'MEDIUM',
  COACH_STUDENT_ATTENDANCE_CHANGED: 'LOW',
  COACH_MEMBER_STATUS_CHANGED: 'LOW'
}

function handleCoachEvent(event: CoachWSEvent) {
  const priority = WS_EVENT_PRIORITY[event.type]
  
  // HIGH优先级：震动+弹窗提醒（学员取消课程属于紧急事件）
  if (priority === 'HIGH') {
    uni.vibrateShort()
    uni.showModal({
      title: '学员取消预约',
      content: `${event.memberName} 取消了 ${event.courseName}`,
      showCancel: false
    })
    refreshCurrentPage()
  }
  
  // MEDIUM优先级：Badge提示（不打断当前操作）
  if (priority === 'MEDIUM') {
    showBadge('students', '+1')
    // 仅当用户在学员列表页时才自动刷新
    if (getCurrentRoute() === '/pages/coach/student-list') {
      refreshCurrentPage()
    }
  }
  
  // LOW优先级：静默刷新（仅当用户在相关页面时）
  if (priority === 'LOW') {
    if (getCurrentRoute().includes('student')) {
      refreshCurrentPage()
    }
  }
}

// 防抖合并（200ms内同一coachId的多个事件合并处理，防止推送风暴）
const debouncedRefresh = debounce(refreshCurrentPage, 200)
```

### 6.2 测试矩阵（最小必测）
#### 后端单元 / 集成
- **越权用例**：A 教练伪造 B 教练学员的 memberId 调 5 个 `{id}` 端点 → 必须 403
- **出勤率边界**：学员只有 0 条预约 → 出勤率 = 0 / NULL，接口不抛 500
- **预约状态码兼容**：`status=4` 必须显示为“已完成”，不得显示为“缺席”
- **考勤状态统计**：`attendance_status=2` 计入完成，`attendance_status=3` 计入缺席；取消/待支付/已支付/进行中不进入出勤率分母
- **考勤写入链路**：教练签到后 `attendance_status=1` 且 `actual_checkin_time` 有值；教练完成后 `attendance_status=2` 且 `actual_finish_time` 有值；登记缺席后 `attendance_status=3` 且不误写为 `status=4`
- **历史回填**：旧数据 `status=4` 且 `attendance_status` 为空/0 → 回填为 `attendance_status=2`；取消记录不得自动回填为缺席
- **关键词搜索**：姓名 / 手机号同时命中；空串 / 空格 keyword 行为与现 `findAll` 一致
- **分页**：page<1、size>100 的降级逻辑与现 `findAllForCoach` 一致
- **参数防护**：非法 `memberId/status/attendanceStatus/orderBy/dateRange` 必须在 Controller/Service 层拦截，不得进入 Mapper
- **缓存有效性**：连续 2 次访问同一 `{coachId}:{memberId}` 关系，第二次应命中 `CoachStudentRelationService` 缓存；测试覆盖 Spring 代理调用而非同类自调用
- **消费记录边界**：教练只能看到与自己课程相关的消费记录；不得返回该学员其他场馆消费、充值明细、全量余额流水

#### 前端验证
- TS 类型检查（项目现 `type-check-errors.txt` 同工具）：**不得新增 error**
- 头像 404 场景：
  1. 手动将接口 avatar 字段改成 `null` → 必须显示默认 SVG
  2. 手动将 avatar 改成 `https://invalid-url.com/404.jpg` → `@error` 触发后显示默认 SVG
  3. 默认 SVG 本身不存在 → 组件应有最终兜底（纯色占位符）
- 手机号脱敏：
  - 11 位数字（`13812341001`）→ `138****1001`
  - 非 11 位（虚拟号 `95123456`）→ 原样返回 `95123456`
  - 空值 → 显示 `未填写`
- 拨打电话确认弹窗：
  1. 点击"一键拨打"按钮 → 弹窗显示完整号码
  2. 确认后 → 调用 `uni.makePhoneCall`
  3. 取消 → 无任何操作

## 8. 关键风险与回滚预案

| 风险 | 触发条件 | 影响 | 回滚 / 预案 |
|------|---------|------|-------------|
| Mapper 新 SQL 在生产数据量下慢查询 | 某热门教练学员 > 5000、时间范围拉得太大 | 接口 5xx，教练工作台连带卡住 | 1. 严格分页，默认 size=20（列表），size=50（详情Tab），最大 size=100；2. P1 的考勤字段与关键索引必须先上（DBA 确认 EXPLAIN 结果）；3. 慢日志阈值设 500ms，超阈值立即降级到「仅查近 90 天预约」；4. 使用 `idx_cb_member_attendance` + `idx_c_coach_date_range` 加速出勤率和最近上课计算 |
| 预约状态码与考勤状态混用 | 开发者继续把 `status=4` 当“缺席”或用 `status=3` 当“已完成” | 出勤率、风险学员、时间线全部错误 | 1. Mapper 注释和 TS 类型必须同时写明两套状态；2. 单测覆盖 `status=4` 显示“已完成”；3. Code Review 检查所有缺席统计只能读取 `attendance_status=3` |
| 新增考勤字段没人维护 | 只建字段和查询，未改 `updateStatusForCoach` / Mapper 更新方法 | 新接口长期显示 0 出勤或历史数据正常、新数据异常 | 1. P1 必须同时改实体、Mapper、Service 状态写入；2. 签到/完成/缺席测试纳入 M1；3. 监控 `status=4 AND attendance_status<>2` 的异常计数 |
| 缓存注解因同类自调用失效 | `ensureCoachCanViewStudent` 和 `canCoachViewStudent` 放同一个 Service 内部调用 | 高频详情接口仍反复查库，缓存穿透防护失效 | 1. 独立 `CoachStudentRelationService` 承载 `@Cacheable`；2. 集成测试验证第二次命中缓存；3. 监控关系校验 SQL QPS |
| 越权逻辑漏洞 | `ensureCoachCanViewStudent` 被后续 refactor 误删或改条件 | 教练可查任意会员（含跨场馆/非学员） | P2 写越权单测并 CI 必跑；PR checklist 单独立一条「所有 {id} 接口首行调用 ensureCoachCanViewStudent」；**缓存失效机制**必须同步测试 |
| 越权校验缓存穿透 | 恶意用户频繁请求不存在的 memberId | 缓存无效，每次都查库，DB 压力飙升 | 1. 对不存在的关系也缓存（值为 `false`），TTL=5min；2. 接口层加限流（同一 coachId 每秒最多查询10个不同 memberId）；3. 监控 `isStudentTaughtByCoach` 的 cache miss 率，超 30% 报警 |
| 消费/充值隐私越界 | 直接复用通用 `MemberService#getConsumeRecords` 或充值接口 | 教练看到与自己无关的消费、充值流水或账户隐私 | 1. 新增教练范围消费查询，仅返回当前教练课程相关记录；2. 充值记录默认不开放；3. 隐私字段需 P0 单独确认 |
| DDL 锁表或回填失败 | 生产 `biz_course_booking` 数据量大，直接 ALTER/UPDATE | 下单、签到、支付链路阻塞 | 1. DBA 审核 online DDL；2. 维护窗口或分批迁移；3. 回填按 id 分批并记录断点；4. 应用回退时保留新增字段 |
| 前端头像 404 成破图 | sys_user.avatar 字段为空、或旧数据 URL 过期、或 CDN 故障 | 学员列表全是破图图标，影响专业度 | **三层兜底**：1. Service 层空值替换为常量；2. 工具函数 `getStudentAvatarWithFallback` 兜底；3. `@error` 事件兜底常量；PR6 人工回归 3 种 404 场景 |
| 训练进度"按课程名聚合"被误解为"课程包 1/N" | 教练以为进度条是其购买的 10 节私教课包 | 客诉：显示进度与收费合同不符 | Tab2 页脚强制加**醒目说明卡**（橙色背景 + info 图标）：「进度基于课程名称自动聚合，非正式课程包。如需 1v1 课程包请联系管理员」；为方案 B（正式课程包）保留后续迭代入口 |
| WebSocket 推送风暴 | 批量变更预约状态时一次性推太多事件（如教练批量取消 50 节课） | 教练端 UI 抖动 / 性能下降 / 弹窗轰炸 | 1. 后端事件按 `coachId` 维度做 **200ms 窗口合并**（同一教练的多个事件合并为 1 条）；2. 前端订阅处**防抖 200ms**；3. HIGH 优先级弹窗**1分钟内最多1次**（防止连续取消导致弹窗轰炸）；4. 前端仅刷新当前可见页的局部 state，非可见页延迟到 onShow 刷新 |
