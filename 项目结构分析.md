# BMP 项目结构分析

> 复核日期：2026-04-25  
> 复核口径：以当前仓库实际目录、源码入口和构建方式为准

## 一、根目录现状

```text
BMP/
├─ .codex/
├─ .github/
├─ .idea/
├─ .vscode/
├─ BMP-uniapp/
├─ logs/
├─ src/
├─ target/
├─ uploads/
├─ vue/
├─ pom.xml
├─ README.md
├─ API文档.md
├─ 数据库设计文档.md
├─ 模块化测试文档.md
├─ 项目分析总结.md
├─ 项目开发文档.md
└─ 项目结构分析.md
```

当前仓库是典型的“后端 + Web + UniApp”三段式单仓结构，没有在根目录发现 `BMPandroid/`。

## 二、目录职责

### 1. `src/`

后端 Spring Boot 工程，是真正的业务中心。

关键子目录：

- `src/main/java/com/badminton/bmp/common`
- `src/main/java/com/badminton/bmp/config`
- `src/main/java/com/badminton/bmp/framework`
- `src/main/java/com/badminton/bmp/modules`
- `src/main/java/com/badminton/bmp/websocket`
- `src/main/resources`

### 2. `vue/`

Vue 3 Web 工程。

它不是单纯后台，而是同时承载：

- 官网入口
- 管理端
- 用户端
- 教练端

### 3. `BMP-uniapp/`

UniApp 工程。

它不只是普通用户移动端，当前还包含：

- 用户业务页面
- `pages/president/**` 会长移动管理端
- 独立 API 层
- Pinia 状态层
- 若干测试文件

### 4. `uploads/`

运行期上传资源目录，和头像、场馆图、器材图等能力直接相关。

### 5. `logs/`

运行期日志目录。

### 6. `target/`

Maven 构建产物目录。

## 三、后端结构

### 1. 业务模块

`src/main/java/com/badminton/bmp/modules/` 下当前有 15 个业务域：

- `activity`
- `booking`
- `coach`
- `course`
- `court`
- `dashboard`
- `equipment`
- `finance`
- `member`
- `notification`
- `search`
- `stringing`
- `system`
- `tournament`
- `venue`

### 2. 典型组织方式

每个业务域基本遵循：

- `controller`
- `service`
- `service/impl`
- `mapper`
- `entity`

个别模块会额外包含：

- `dto`
- `vo`
- `cache`

这说明后端已经形成了比较统一的模块组织规范。

### 3. 横切层

后端横切能力集中在：

- `config/`：安全、缓存、Swagger、调度、Redis、上传、JWT Filter 等
- `common/`：返回体、JWT、异常、校验、工具类
- `websocket/`：STOMP 鉴权、Broker 配置、推送服务

## 四、Web 端结构

### 1. 核心目录

```text
vue/src/
├─ api/
├─ components/
├─ composables/
├─ config/
├─ directives/
├─ layout/
├─ router/
├─ styles/
├─ utils/
└─ views/
```

### 2. 结构含义

- `layout/`：按角色和场景拆分布局
- `router/index.js`：整个 Web 端的角色分流总入口
- `views/`：管理端、用户端、教练端和站点页面
- `api/`：与后端 REST 接口对接
- `components/charts/`：Dashboard 图表组件较多，说明看板能力占比较高

### 3. 一个结构特点

`vue/` 把官网与后台放在同一个工程里，这会带来两个结果：

- 优点：共享鉴权、样式、构建链路，维护入口集中
- 风险：依赖更重、打包更大、职责边界更容易变宽

## 五、UniApp 端结构

### 1. 核心目录

```text
BMP-uniapp/
├─ api/
├─ components/
├─ composables/
├─ config/
├─ directives/
├─ pages/
├─ scripts/
├─ static/
├─ store/
├─ styles/
├─ tests/
├─ types/
├─ utils/
├─ main.ts
├─ pages.json
└─ manifest.json
```

### 2. 结构特点

- `pages/` 体量已经很大，说明业务页面覆盖度高
- `api/president/*` 与 `pages/president/*` 成体系存在，说明移动管理端不是临时拼接
- `tests/` 已经开始承担部分基础逻辑验证
- 同时存在根部 `pages.json/manifest.json` 与 `src/pages.json/src/manifest.json`，属于需要持续留意的配置双源现象

### 3. 角色边界

从结构上看，UniApp 端至少有两层入口：

- 普通用户业务入口
- 会长移动管理入口

这也是当前项目结构里最值得注意的演化方向。

## 六、当前结构的优点

### 1. 多端边界总体清晰

后端、Web、UniApp 各自独立目录，构建方式分开，整体没有互相缠绕到不可维护。

### 2. 后端模块划分稳定

业务域清晰，横切层清晰，属于适合继续迭代的模块化单体。

### 3. UniApp 已经具备独立工程形态

不是“把 Web 页面改成小程序页面”那种轻量适配，而是有自己的状态、工具层、页面体系和测试。

## 七、当前结构的风险点

### 1. 根目录文档较多，信息容易漂移

分析类、开发类、测试类文档都放在根目录，长期维护时很容易出现多份文档并行失真。

### 2. Web 端职责偏多

官网、管理端、用户端、教练端共处一个工程，业务越长越大，构建和依赖治理成本就越高。

### 3. UniApp 配置存在双源痕迹

根部与 `src/` 下都存在 `pages.json/manifest.json`，如果后续多人协作，需要明确哪一套是最终事实来源。

## 八、结构层面的建议

### 建议 1

把根目录文档逐步收敛，只保留少量总入口文档，其他转到单独 `docs/` 目录。

### 建议 2

后续如果 Web 端继续膨胀，评估是否把官网从后台工程中拆出，减轻主工程包体积和职责压力。

### 建议 3

对 UniApp 的配置来源做一次确认，避免根部和 `src/` 双份配置长期并存。

## 九、结论

当前 BMP 的结构本身没有明显“必须重构”的硬伤。

它的问题不在于目录混乱，而在于：

- 多端都已经做起来了
- 结构还能承受
- 但文档、角色口径、配置来源和前端治理需要一起跟上

所以从结构角度看，当前项目最合理的策略不是推翻重来，而是**在现有单仓多端格局上做收敛和治理**。
