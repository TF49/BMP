## 1. 系统概览

### 1.1 项目定义

羽擎（Badminton Management Platform，BMP）是面向羽毛球场馆经营场景的单仓多端业务系统，**v1.0.0 功能开发已完成**，当前同时覆盖：

- 场馆、场地、会员、教练、课程、赛事、器材等基础经营资源
- 预约、租借、报名、穿线、充值、消费、财务等交易与服务流程
- Web 官网、Web 管理端、Web 用户端、Web 教练端
- UniApp 用户端、移动管理端、教练端（微信小程序）

### 4.3 教练学员与考勤实现

教练学员能力不维护独立关系表，而是从 `biz_course_booking.member_id → biz_course.coach_id` 推导：

1. 仅 `COACH` 可访问 `/api/coach/students/**`，并通过当前登录账号绑定的 `sys_coach.user_id` 确定教练身份。
2. 教练只能查看与本人课程有关、课程结束时间距今不超过 180 天的有效学员关系。
3. 列表支持关键字、会员类型、风险学员、今日学员、排序和分页；手机号在查询层脱敏。
4. 风险口径默认为出勤率低于 70%，或最近完成课程距今超过 14 天；对应配置为 `coach.risk-threshold.*`。
5. 考勤通过 `CHECK_IN → COMPLETE` 或 `ABSENT` 状态机更新，单教练限制为每秒 5 次考勤命令。
6. 查询限流为列表每教练每秒 5 次、详情类每教练每秒 10 次；数据库异常时优先返回 15 分钟内的最后成功结果，否则返回带 `traceId` 的 503 结果。
7. 关系、训练进度、出勤缓存 TTL 为 5 分钟，最后成功结果 TTL 为 15 分钟；课程预约变化会触发关系缓存失效。

## 5. 多端结构

### 5.1 Web 端

| 场景 | 入口 | 角色 |
|------|------|------|
| 官网 | `/site` | 公开 |
| 管理端 | `/dashboard`、`/venue/*`、`/finance/*` 等 | `PRESIDENT`、`VENUE_MANAGER` |
| 用户端 | `/user/*` | `USER`、`MEMBER` |
| 教练端 | `/coach/*` | `COACH` |

路由守卫要点：

1. `ADMIN` 映射为 `PRESIDENT`
2. `USER` / `MEMBER` 访问管理路由时跳转用户端
3. `COACH` 访问用户端或管理端时跳转教练端

### 5.2 UniApp 端

顶层页面目录（17 组）：`booking`、`coach`、`course`、`equipment`、`index`、`login`、`notice`、`president`、`profile`、`recharge`、`recover`、`register`、`search`、`settings`、`stringing`、`tournament`、`venue`。

| 页面组 | 目录 | 说明 |
|--------|------|------|
| 用户与公共 | `pages/index`、`venue`、`booking` 等 | 用户主链路 |
| 移动管理 | `pages/president/**` | 全业务移动管理 |
| 教练 | `pages/coach/**` | 工作台、课程、课表、预约、学员 |

当前教练学员列表/详情和原生 STOMP 客户端落在 UniApp；Web 教练路由当前仍是工作台、课程、课表、预约明细和档案，不应把 UniApp 学员页误记为 Web 已交付页面。

## 8. 实现边界与约定

### 8.1 后端

- 已有 13 个后端测试类，集中覆盖支付自动取消、教练学员权限/隐私/降级、课程考勤和 WebSocket；其他业务域仍需 Swagger 与角色化联调回归
- 近期动态由 `member`、`booking`、`finance` 聚合，**无** `biz_activity` 表
- `payment` 模块提供自动取消只读配置，扫描逻辑由调度任务执行
- 官网 `GET /api/site/overview` 是匿名汇总接口，只返回场地利用率、预约成功率、器材可用数和当日收入等展示数据

### 8.2 Web

- 单工程承载四端，生产构建需关注分包与按需加载
- 开发代理：`vue.config.js` → `http://localhost:9090`

### 8.3 UniApp

- 与 Web 共用 JWT 与 `/api` 前缀（通过 `config` 配置基址）
- 微信小程序需配置合法域名与腾讯地图 Key（填地址场景）
- WebSocket 使用 `/ws-native` 原生端点，STOMP `CONNECT` 头传递 JWT，并订阅 `/user/queue/notifications`
- 当前安全白名单只显式放行 `/ws` 与 `/ws/**`，没有放行 `/ws-native`；小程序实时推送上线前需要补齐握手规则并真机验证

### 8.4 配置与运行环境

- 默认 profile 为 `dev`，端口 `9090`；开发环境 WebSocket 来源默认 `*`
- 开发环境未配置 Redis 时使用 Caffeine；`prod` profile 要求 `REDIS_HOST`，并启用 Redis Cache
- 支付自动取消开发默认 `5` 秒 / `1000` 毫秒扫描，生产默认 `900` 秒 / `60000` 毫秒扫描
- `prod` profile 禁用 Swagger；数据库、JWT、天气、地图和 WebSocket 来源通过环境变量覆盖

### 8.5 编码与 Git 约定

- 后端：按域拆分 `controller / service / mapper / entity`
- 前后端权限口径一致，尤其 `PRESIDENT` / `VENUE_MANAGER` / `COACH`
- 提交信息：`feat` `fix` `docs` `refactor` `perf` `test` `chore`
