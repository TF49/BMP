# 支付功能 5 分钟未支付自动取消现状评估、差距清单与修正顺序

## 1. 文档信息

- 更新日期：2026-05-18
- 文档性质：现状评估与修正基线
- 目标：基于当前仓库真实开发进度，明确该功能已完成内容、当前差距以及后续修正顺序

## 2. 评估范围

本次评估仅覆盖以下 5 类业务订单：

1. 场地预约
2. 课程预约
3. 器材租借
4. 赛事报名
5. 穿线服务

明确不包含：

- 会员充值

## 3. 当前现状评估

### 3.1 公共基础能力现状

当前仓库中，支付超时自动取消的公共能力已经不是“待实施”，而是已经落地了基础骨架：

1. 已有独立配置类 `PaymentAutoCancelProperties`
   - 支持 `enabled`
   - 支持 `timeoutSeconds`
   - 支持 `scanIntervalMs`
   - 兼容旧配置 `timeoutMinutes`

2. 已有只读配置接口 `GET /api/payment/auto-cancel/config`
   - 已返回 `enabled`
   - 已返回 `timeoutMinutes`
   - 已返回 `timeoutSeconds`
   - 已返回 `serverTime`

3. 已有统一定时任务入口 `ScheduleConfig.autoCancelExpiredUnpaidOrders()`
   - 已按模块顺序执行
   - 已对单模块做独立 `try/catch`
   - 已支持配置开关控制是否启用

4. 已有统一超时判断基准
   - 后端以 `create_time` 为超时依据
   - 支付入口已在支付前再次判断是否超时
   - 支付 SQL 已使用条件更新，避免超时单被继续支付

结论：

- 公共骨架已存在
- 文档不应再把这些内容描述为“建议新增”

### 3.3 前端现状

前端部分也不是“未开始”，而是已经形成共享能力并接入多处页面：

1. `vue` 端已存在共享倒计时组合式能力
   - 已封装 `usePaymentAutoCancel`
   - 已封装倒计时文案与到期刷新逻辑
   - 已接入用户端与管理端多个订单页面

2. `BMP-uniapp` 端已存在对应共享能力
   - 已封装 `usePaymentAutoCancel`
   - 已根据服务端配置计算剩余支付时间
   - 已接入用户端和 president 端多个订单页面

3. 前端当前实现特征
   - 倒计时以服务端配置为准
   - 会使用 `serverTime` 进行本地校时
   - 超时后会主动刷新列表或详情

结论：

- 前端不应再按“从零补充倒计时展示”的口径描述
- 更准确的说法应为“已有共享倒计时能力，需校准配置与做覆盖验证”

### 3.4 当前实现与目标之间的关键偏差

当前最需要关注的，不是“有没有做”，而是“做出来的默认行为是否符合原目标”。

#### 3.4.1 默认超时时长与目标不一致

原目标是：

- 5 分钟未支付自动取消

当前默认配置却是：

```properties
bmp.payment.auto-cancel.timeout-seconds=${BMP_PAYMENT_AUTO_CANCEL_TIMEOUT_SECONDS:5}
```

这表示当前默认值更接近：

- 5 秒超时

该项属于最高优先级偏差。

#### 3.4.2 默认扫描频率与目标不一致

原目标是：

- 每分钟扫描一次

当前默认配置却是：

```properties
bmp.payment.auto-cancel.scan-interval-ms=${BMP_PAYMENT_AUTO_CANCEL_SCAN_INTERVAL_MS:1000}
```

这表示当前默认值更接近：

- 每 1 秒扫描一次

该项同样属于最高优先级偏差。

#### 3.4.3 文档配置口径与代码口径不一致

原文档主要使用：

```properties
bmp.payment.auto-cancel.timeout-minutes=5
```

当前代码主口径已经变为：

```properties
bmp.payment.auto-cancel.timeout-seconds=...
```

并通过兼容逻辑保留了 `timeoutMinutes`。

这意味着：

- 文档口径与代码主配置口径已经分叉
- 后续维护者容易误判真实配置项

#### 3.4.4 历史旧单首次清理只写了目标，没写执行边界

当前实现确实会在定时任务运行时清理已超时旧单，但文档仍缺少以下关键落地说明：

- 是否需要分页或分批处理
- 单次扫描允许处理的数量上限
- 出错后的补跑策略
- 需要观察哪些日志指标
- 首次上线时是否需要人工确认历史数据量

#### 3.4.5 未支付状态的历史兼容性未在文档中体现

当前 SQL 普遍兼容：

- `payment_status IS NULL`
- `payment_status = 0`

说明项目中实际上存在“未支付”双语义兼容。

文档如果只写“未支付就是 0”，会掩盖当前真实历史数据状态。

#### 3.4.6 自动化测试要求仍然偏弱

当前文档仍以编译检查和手工回归为主，但对以下高风险点约束不足：

- 支付与自动取消并发竞争
- 库存/人数/场地状态的幂等副作用
- 配置关闭时的行为
- 首次扫描历史旧单的清理效果

## 4. 差距清单

### 4.1 P0：必须优先修正

#### 差距 1：默认超时时长错误

- 现状：默认值为 5 秒
- 目标：默认值应为 5 分钟
- 风险：测试与实际业务规则完全偏离，前后端倒计时表现都会失真

#### 差距 2：默认扫描频率错误

- 现状：默认值为 1 秒扫描一次
- 目标：默认值应为 1 分钟扫描一次
- 风险：无意义增加数据库扫描频率，放大日志与任务开销

#### 差距 3：文档已落后于真实代码进度

- 现状：文档仍按“准备新增”口径描述
- 目标：文档应切换为“已完成现状 + 待修正差距”
- 风险：后续实施、联调、回归判断基线失真

### 4.2 P1：应在默认配置修正后尽快补齐

#### 差距 4：配置主口径未统一

- 现状：文档主写 `timeout-minutes`，代码主用 `timeout-seconds`
- 目标：统一一套主配置口径，其余仅保留兼容说明
- 风险：环境变量、配置中心、运维文档不一致

#### 差距 5：历史旧单清理策略缺少操作级说明

- 现状：只有“首次扫描会清理”的目标性描述
- 目标：补充数据量评估、日志观察点、失败补跑说明
- 风险：上线首次扫描不可观测，问题定位困难

#### 差距 6：缺少针对高风险链路的自动化验证

- 现状：以手工回归为主
- 目标：补最小必要的服务层/集成测试
- 风险：并发竞争和幂等问题只能上线后暴露

### 4.3 P2：建议在功能稳定后整理

#### 差距 7：`payment_status` 历史兼容规则未文档化

- 现状：代码兼容 `NULL/0`
- 目标：文档明确该兼容事实，并决定未来是否清洗历史数据

#### 差距 8：接口返回格式说明不够精确

- 现状：文档示例与当前 `serverTime` 返回格式不完全一致
- 目标：文档与真实接口格式保持一致

## 5. 修正顺序

### 第 1 步：当前阶段暂不调整默认配置基线

当前决定：

- 开发联调阶段继续保留 `5 秒超时 + 1 秒扫描`
- 暂不立即把默认值切回 `5 分钟超时 + 1 分钟扫描`

原因：

1. 当前需要快速验证倒计时显示、自动取消链路和并发竞争
2. 使用 `5 秒/1 秒` 可以显著缩短联调等待时间
3. 该配置目前作为开发阶段基线保留，不作为最终发布配置结论

说明：

- 本文档当前阶段直接开始执行第 2、3、4、5 步
- 第 1 步顺延到提测前或发布前再处理
- 届时再将默认值调整为真实业务目标值

### 第 2 步：统一文档与配置口径

在保留开发期 `5 秒/1 秒` 配置的前提下，统一以下内容：

1. 本文档
2. `README.md` 中涉及该功能的说明
3. 若有接口文档，也同步更新
4. 明确开发期配置与发布期配置的区分方式
5. 明确主配置项以 `timeout-seconds` 还是 `timeout-minutes` 为准

建议：

- 代码保留兼容逻辑
- 文档只保留一套主配置写法
- 文档中明确标注当前开发联调基线为 `5 秒超时 + 1 秒扫描`
- 文档中明确标注提测/生产目标为 `5 分钟超时 + 1 分钟扫描`

### 第 3 步：补充高风险测试与验收项

建议把验收重点改为以下几项：

1. 5 个业务模块分别验证超时自动取消是否生效
2. 支付与自动取消并发竞争时，只允许一方成功
3. 器材租借库存回补不能重复执行
4. 课程与赛事人数回滚不能重复执行
5. 场地预约取消后场地状态必须正确重算
6. 配置关闭后定时任务不应执行取消逻辑
7. 历史超时旧单在首次扫描后应被清理
8. 倒计时文案、剩余秒数变化和过期后的状态刷新必须一致

当前阶段优先测试口径：

- 以开发联调配置 `5 秒超时 + 1 秒扫描` 为主
- 优先验证“倒计时每秒变化是否正常”
- 优先验证“超时后是否在短时间内自动刷新为已取消”

### 第 4 步：做一次按模块的回归验证

建议仍按模块渐进回归，但回归重点应从“有没有做”切换为“当前实现是否符合目标”：

1. 场地预约
2. 课程预约
3. 器材租借
4. 赛事报名
5. 穿线服务

每个模块至少验证：

- 待支付订单创建后倒计时是否正常
- 倒计时显示是否按秒更新，而不是 5 秒一跳
- 超时后是否自动变更为已取消
- 超时后是否禁止继续支付
- 业务副作用是否只执行一次

### 第 5 步：最后整理低优先级一致性问题

在主链路验证通过后，再处理以下整理项：

1. 是否保留 `timeoutMinutes` 兼容逻辑
2. 是否补充历史 `payment_status = NULL` 的数据清洗方案
3. 是否进一步收敛日志与监控字段

## 7. 结论

当前项目中，“支付 5 分钟未支付自动取消”能力已经基本开发完成，问题重点已经不再是“是否具备实现方案”，而是：

1. 文档是否真实反映当前开发阶段策略
2. 高风险链路是否完成足够验证
3. 倒计时显示与过期刷新是否一致
4. 提测前是否完成配置基线切换

因此，后续推进不应继续按“从零实施”的思路展开，而应按以下策略执行：

- 当前阶段先跳过默认配置切换
- 直接推进文档统一、测试补齐、模块回归和一致性整理
- 在提测前或发布前再完成默认配置基线切换

按这个顺序推进，风险最小，也最符合当前仓库的真实开发状态。
