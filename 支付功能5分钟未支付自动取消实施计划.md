# 支付功能 5 分钟未支付自动取消实施计划

## 1. 文档信息

- 生成日期：2026-05-17
- 计划实施开始日期：2026-05-18
- 目标：在不破坏当前项目整体稳定性的前提下，为业务订单增加“创建后 5 分钟未支付自动取消”能力

## 2. 实施范围

本次仅覆盖以下 5 类业务订单：

1. 场地预约
2. 课程预约
3. 器材租借
4. 赛事报名
5. 穿线服务

明确不包含：

- 会员充值

## 3. 方案摘要

- 以后端为唯一真相源，新增统一的自动取消能力。
- 超时判断基于订单 `create_time`，不是 `update_time`。
- 规则为：订单创建后超过 5 分钟，且仍处于“待支付/未支付”状态，则自动取消。
- 上线后第一次定时扫描时，立即清理历史上已经超时的未支付旧订单。
- 前端同步补充倒计时展示，覆盖 `vue` 管理端和 `BMP-uniapp` 用户端。
- 不新增退款记录、不新增财务记录、不改数据库表结构。

## 4. 配置设计

为降低上线风险，本次不写死配置，新增如下配置项：

```properties
bmp.payment.auto-cancel.enabled=true
bmp.payment.auto-cancel.timeout-minutes=5
```

默认策略：

- 默认开启
- 默认超时时长为 5 分钟

建议同时新增一个只读配置接口，供两套前端获取：

```http
GET /api/payment/auto-cancel/config
```

返回建议：

```json
{
  "enabled": true,
  "timeoutMinutes": 5,
  "serverTime": "2026-05-17T12:00:00"
}
```

## 5. 后端实施方案

### 5.1 定时任务

在现有 `ScheduleConfig` 中新增 1 个每分钟执行的自动取消任务。

执行要求：

- 每分钟扫描一次
- 按模块顺序依次处理
- 每个模块单独 `try/catch`
- 单个模块失败不能影响其他模块继续执行
- 输出取消数量和异常日志

### 5.2 服务层实现原则

不要直接复用当前面向用户/管理员的公开取消入口，因为其中包含登录态和角色权限校验，定时任务上下文下容易被拦截。

推荐为每个业务服务新增内部方法，例如：

```java
int autoCancelExpiredUnpaidOrders(LocalDateTime cutoff)
```

处理逻辑统一为：

1. 查询候选记录
2. 仅筛选：
   - `del_flag = 0`
   - `payment_status = 0`
   - 业务状态仍为待支付
   - `create_time <= cutoff`
3. 更新前再次校验数据库状态
4. 更新为已取消
5. 执行该模块现有的库存、人数、场地状态、通知等副作用

### 5.3 数据更新原则

自动取消时统一遵循：

- `status -> 0`
- `payment_status` 保持 `0`
- `update_time -> now`

注意：

- 不能改成退款中或已退款
- 因为这类订单根本没有实际支付成功

### 5.4 并发与幂等

需要重点防止“4 分 59 秒支付”和“5 分钟定时取消”同时发生的竞争问题。

处理原则：

- 支付和自动取消都必须依赖带条件的数据库更新
- 谁先成功，谁生效
- 自动取消后，支付接口继续按现有逻辑返回“已取消，无法支付”
- 自动取消逻辑必须幂等，多次执行不能重复扣减或重复回补

## 6. 各模块副作用要求

### 6.1 场地预约

- 取消未支付订单后，必须重算相关场地状态
- 继续复用现有订单状态通知和待办刷新逻辑

### 6.2 课程预约

- 取消未支付订单后，必须减少课程当前报名人数
- 需要同步清理课程相关缓存
- 保留现有 WebSocket 推送与 dashboard 刷新逻辑

### 6.3 器材租借

- 该模块风险最高
- 因为当前未支付订单创建后就可能已经占用了库存
- 自动取消后，必须正确回补器材可用库存
- 保留现有订单状态通知逻辑

### 6.4 赛事报名

- 自动取消后，必须重算赛事当前占位人数
- 保留现有订单状态推送和 dashboard 刷新逻辑

### 6.5 穿线服务

- 自动取消后更新状态并推送通知即可
- 不生成退款、财务、消费冲正记录

## 7. 前端实施方案

### 7.1 总体原则

- 后端状态为准
- 倒计时是体验增强，不是业务真相
- 如果配置接口失败，前端不展示倒计时，但现有支付与状态流转仍可继续工作

### 7.2 覆盖范围

本次前端倒计时同时覆盖：

- `vue` 管理端
- `BMP-uniapp` 用户端

### 7.3 展示策略

详情页、确认页、支付页：

- 展示秒级倒计时 `mm:ss`

列表页、表格页：

- 展示剩余时间标签
- 避免逐秒重绘全列表，建议按可见数据做低频刷新

### 7.4 到期行为

当前端计算剩余时间小于等于 0 时：

- 立即禁用支付按钮
- 展示“订单超时，正在刷新”或等效提示
- 主动刷新订单详情或列表
- 最终以服务端返回的真实状态为准

## 8. 验收与测试

### 8.1 后端验收

必须覆盖以下场景：

1. 场地预约超时未支付后自动取消，场地占用释放
2. 课程预约超时未支付后自动取消，报名人数正确回滚
3. 器材租借超时未支付后自动取消，库存正确回补
4. 赛事报名超时未支付后自动取消，占位人数正确回滚
5. 穿线服务超时未支付后自动取消，不生成退款记录
6. 历史超时旧单在功能上线后首次扫描时被清理
7. 支付与自动取消并发竞争时，只允许一方成功

### 8.2 构建检查

建议实施完成后执行：

```bash
mvn -q -DskipTests compile
```

`vue` 端建议执行：

```bash
npm run build
```

`BMP-uniapp` 端建议执行：

```bash
npm run type-check
```

如有需要，可为倒计时工具补充 `vitest` 用例。

### 8.3 手工回归

建议按以下顺序回归：

1. 创建待支付订单
2. 观察倒计时递减
3. 到期后自动刷新状态
4. 到期后支付按钮禁用
5. 管理端与用户端状态保持一致

## 9. 实施顺序建议

建议按以下顺序推进：

1. 后端配置项
2. 定时任务骨架
3. 5 个业务模块的内部自动取消能力
4. 配置查询接口
5. `vue` 管理端倒计时展示
6. `BMP-uniapp` 用户端倒计时展示
7. 编译检查与回归验证

## 10. 结论

该功能可以在当前项目架构下较稳妥地落地，前提是：

- 不把充值纳入本期范围
- 不直接调用带权限校验的公开取消入口
- 严格复用各模块已有的库存、人数、场地状态、通知副作用
- 用配置化与分钟级定时任务控制上线风险

按本计划执行，适合从 2026-05-18 开始进入实施阶段。
