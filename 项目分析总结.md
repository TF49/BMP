# 羽擎（BMP）项目分析总结

> 分析日期：2026-04-25  
> 分析方式：仅基于当前仓库中的代码、配置、SQL、构建结果与测试结果复核，不以既有分析类 Markdown 为依据  
> 项目路径：`D:\Developer Tool\IDEA\JAVAEE\BMP`

## 一、总体判断

这不是一个“单端应用”，而是一个已经进入中后期的单仓多端业务系统：

- `Spring Boot` 后端承担统一业务核心
- `vue/` 承担官网、后台管理、用户端、教练端四类 Web 场景
- `BMP-uniapp/` 承担普通用户移动端，同时已经扩展出 `president` 移动管理端

如果用一句话概括当前项目状态，就是：

**业务覆盖面已经很完整，工程规模也已经不小，但跨端角色策略、前端体积控制、文档一致性和后端测试体系还没完全跟上。**

## 二、代码层面复核出的真实规模

### 1. 后端

- Java 源文件：`174` 个
- 业务模块：`15` 个
- 控制器入口：约 `20+`
- 配置覆盖：安全、缓存、Redis、JWT、Swagger、WebSocket、定时任务、上传等

后端不是简单 CRUD 组合，已经有较明显的“业务编排层”特征：

- 支付/退款
- 冲突检测
- 状态自动流转
- 数据权限兜底
- WebSocket 实时推送
- 财务流水联动

### 2. Web 端

- 主要源码文件（`views/components/api/layout/composables` 范围）：`128` 个
- 路由分层明确
- 后台图表组件数量较多，说明 Dashboard 不是空壳

Web 端的真实定位不是单一后台，而是：

- `SiteLayout`：官网
- `AdminLayout`：会长/场馆管理者后台
- `UserLayout`：普通用户与会员 Web 端
- `CoachLayout`：教练端

### 3. UniApp 端

- `pages/` 下页面文件：`97` 个
- 主要源码文件（排除 node_modules/unpackage 后的核心目录）：`189` 个
- 已包含 `pages/president/**` 的一整套移动管理端页面
- 存在 `Pinia + request + roleCheck + presidentRouter + composables + tests` 这一套基础设施

这说明 UniApp 端已经从“补充端”演进成了真正的业务端，而不是简单展示页。

## 三、架构特点

### 1. 后端是清晰的模块化单体

后端目录结构很稳定：

- `modules/*`：按业务域拆分
- `config/*`：横切配置
- `common/*`：统一返回、JWT、异常、校验、工具
- `websocket/*`：实时推送

它不是微服务，但已经具备了比较成熟的“领域模块 + 横切能力”组织方式。对当前体量来说，这种结构是合理的。

### 2. 权限设计是“多角色 + 多端共用后端”

当前能从代码里直接确认的角色包括：

- `PRESIDENT`
- `VENUE_MANAGER`
- `USER`
- `MEMBER`
- `COACH`

权限控制并非只停留在前端，而是至少分成三层：

- Spring Security 负责入口鉴权
- Service 层通过 `SecurityUtils` 做数据范围控制
- 前端路由守卫和按钮权限做交互层限制

这是当前项目里一个比较成熟的设计点。

### 3. WebSocket 与定时任务是真实接入的

从代码看，WebSocket 不是摆设：

- 后端提供 `/ws` STOMP + SockJS
- 管理端订阅待办和 Dashboard 刷新
- 用户端订阅订单状态变更
- 预约、课程、器材租借、赛事、穿线等服务都存在推送调用

定时任务也已经承担业务职责：

- 完成超时预约
- 推进课程状态
- 推进课程预约状态
- 推进赛事状态
- 推进赛事报名状态
- 标记器材租借逾期
- 清理登录限流器过期记录

这意味着系统已经从“表单管理系统”走到了“持续运行的业务系统”阶段。

## 四、三个端的真实定位

### 1. 后端

后端是统一业务核心，没有明显“为某一个端单独硬编码”的迹象，属于标准共享中台。

### 2. Web 端

Web 端的定位很复杂，但也很完整：

- 外部展示入口：官网
- 内部经营入口：管理端
- 用户自助入口：用户端
- 教练工作入口：教练端

优点是业务闭环完整，缺点是一个工程承载的职责偏多。

### 3. UniApp 端

UniApp 端现在至少承担两套角色体系：

- 普通用户/会员移动端
- 会长移动管理端

这里最值得注意的不是“功能少”，而是**功能已经扩出来了，但角色边界还没有完全收紧**。

## 五、已确认的关键问题

### 1. UniApp 角色定义不一致

`BMP-uniapp/utils/roleCheck.ts` 里：

- 用户端角色统一为 `USER`、`MEMBER`
- 移动管理端角色统一为 `PRESIDENT`、`VENUE_MANAGER`
- `COACH` 在移动端保持未开放状态，并给出明确提示

第一阶段整改后的口径是：

**Web、UniApp、后端统一按同一角色矩阵分流，不再出现 `MEMBER` 或 `VENUE_MANAGER` 被错误拦截的情况。**

### 2. Web 工程包体积已经偏大

本次执行 `vue` 构建通过，但出现大体积告警：

- `chunk-element-plus` 约 `1.0 MiB`
- `chunk-echarts` 约 `896 KiB`
- `chunk-vendors` 约 `776 KiB`
- 主入口总量约 `2.25 MiB`

这不是“构建失败”的问题，而是很典型的“功能不断叠加、工程还没做体积治理”的问题。

### 3. 文档与代码曾经出现过明显漂移

当前仓库根目录没有看到 `BMPandroid/`，但旧 README 曾长期把它当现状描述的一部分。

这说明项目文档维护不是没有做，而是没有完全跟上代码现状。对多端项目来说，这种漂移会直接影响新人理解项目边界。

### 4. 后端测试体系偏弱

当前：

- `pom.xml` 中没有保留测试依赖
- 本次只验证了可编译
- 没看到成体系的后端测试目录和测试基建

这意味着后端复杂业务虽然已经很多，但回归保护主要还是靠人工验证。

- Web 端多角色分流
- UniApp 端基础设施建设

### 还在演进中的部分

## 七、从代码看出的项目气质

这个项目不是“架子先行”，而是明显经历了连续功能迭代：

- 先有完整后端业务域
- 再有 Web 多角色扩展
- 再把移动端从用户端逐步推到管理端

所以它的真实问题不是“没功能”，而是**已经走到要开始治理复杂度的阶段**。

## 十、结论

当前 BMP 的核心判断不是“项目是否成型”，而是：

**它已经成型，而且已经开始进入治理复杂度、统一口径和补测试护栏的阶段。**

如果后续继续投入，最值得优先做的不是再加一个新模块，而是把：

- 多端角色口径
- 前端体积
- 文档一致性
- 后端测试

这四件事补平。这样这个项目才会从“功能很多”真正迈向“长期可维护”。
