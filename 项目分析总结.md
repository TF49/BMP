# 羽擎（BMP）项目分析总结

> 分析日期：2026-05-17  
> 分析口径：以当前仓库代码、构建结果、路由配置、页面注册和根目录文档为准。  
> 本文档只回答三件事：项目现在处在哪个阶段、当前最主要的问题是什么、下一步最该先做什么。

## 一、阶段判断

当前 BMP 已经不是开发初期项目，也不是只搭出壳子的演示工程。

更准确的判断是：

**核心业务已经成型，项目正在进入“联调 + 测试补护栏 + 性能治理 + 文档同步”的阶段。**

这个判断的依据很直接：

- 后端已经稳定存在 `15` 个业务域、`27` 个 Controller
- Web 已同时承载官网、管理端、用户端、教练端四类场景
- UniApp 已注册 `107` 个页面，覆盖用户端、移动管理端、教练端
- 财务审计、财务对账、综合搜索、天气代理、地图逆地理编码、近期动态等增量能力已经进主干

## 二、这轮复核带来的新结论

### 1. 主干可构建性比上轮更好

2026-05-17 的复核结果是：

- `mvn -q -DskipTests compile` 通过
- `vue/` 下 `npm run build` 通过
- `BMP-uniapp/` 下 `npm run type-check` 通过
- `BMP-uniapp/` 下 `npm test` 通过，`4` 个测试文件、`11` 个用例全部通过
- `BMP-uniapp/` 下 `npm run build:mp-weixin` 通过

这说明一个重要变化：

**主干现在的“编译/类型/前端单测/小程序构建”基线是通的。**

换句话说，之前更像“类型债务优先处理”的阶段，现在已经往“质量和性能治理”平移了。

### 2. 项目问题不再是“功能缺口大”

从代码现状看，系统已经具备：

- 用户、会员、充值、消费链路
- 场馆、场地、预约链路
- 教练、课程、课程预约链路
- 器材、租借链路
- 赛事、报名链路
- 穿线、通知、反馈、搜索、天气、地图、财务链路

所以当前最核心的矛盾，已经不是“缺模块”，而是：

**现有模块不少，但自动化测试、性能治理和真实业务回归还没有完全跟上。**

## 三、当前真正需要警惕的问题

### 1. 后端自动化测试仍然明显偏弱

这是当前最硬的一条风险。

虽然 `mvn -q -DskipTests compile` 能通过，但仓库当前没有 `src/test` 目录，说明：

- 高耦合业务模块还缺少回归护栏
- 业务规则变更仍容易依赖人工联调发现问题
- 预约、充值、财务等跨模块联动在后续改动时回归成本较高

这件事的优先级仍然很高。

### 2. Web 工程已经进入包体积治理区间

`npm run build` 虽然通过，但构建输出已经给出非常明确的信号：

- `chunk-element-plus` 约 `1.0 MiB`
- `chunk-echarts` 约 `896 KiB`
- `chunk-vendors` 约 `776 KiB`
- `app` 入口合计约 `2.27 MiB`

这不是“以后也许会变大”，而是已经变大了。

它会直接影响：

- 首屏加载成本
- 构建时间
- 后续继续往单工程里塞功能时的可维护性

### 3. Sass 弃用告警说明技术债已经开始外露

当前 Web 与 UniApp 构建都存在 Sass 旧接口告警。

其中 UniApp 侧还额外暴露了 `uview-plus` 和 `App.vue` 中的 `@import` 弃用警告。它们短期不拦构建，但已经是明确的维护成本信号。

### 4. 多端真实业务回归仍然需要继续补

现在的状态是：

- 编译通过
- 类型检查通过
- 前端单测通过
- 小程序构建通过

但这不等于所有角色关键路径都已经真实走通。

仍然值得继续回归的链路包括：

- 管理端财务审计与对账
- 用户端预约、课程、赛事、器材、穿线支付链路
- 教练端“我的课程 / 课表 / 预约处理”链路
- UniApp 与 Web 在同一业务上的数据一致性

## 四、现在项目最值钱的工作，不是继续铺模块

如果把当前阶段压缩成一句更实的话：

**BMP 现在最值钱的工作，不是再继续铺一个新模块，而是把现有模块的可回归性、可维护性和可交付性补平。**

原因很简单：

1. 功能面已经够广
2. 多端入口已经够多
3. 每增加一个新模块，回归和维护成本都会被继续放大

## 五、下一步优先级建议

### 第一优先级：补后端回归测试

优先覆盖高耦合、跨模块、最容易出财务副作用的链路：

- 场地预约
- 课程预约
- 赛事报名
- 器材租借
- 穿线支付
- 充值与财务联动

### 第二优先级：做 Web 包体积治理

建议重点检查：

- `Element Plus` 的按需引入是否还能继续收紧
- `ECharts` 是否存在整包引入
- 重页面是否继续下沉到路由级拆分
- 公共依赖是否有重复打入多个 chunk

### 第三优先级：清理 Sass 旧接口与 `@import`

这件事不像后端测试那样最急，但应该尽早做，因为它已经跨 Web 和 UniApp 两侧出现。

### 第四优先级：建立角色化回归清单

建议按角色建立固定回归矩阵：

- `PRESIDENT`
- `VENUE_MANAGER`
- `USER`
- `MEMBER`
- `COACH`

让每次较大改动之后，都能快速知道哪些链路必须过一遍。

## 六、结论

当前 BMP 最准确的阶段定位是：

**业务主干已经搭完，项目正在从“能开发”转向“能稳定演进”。**

接下来最应该优先补的，不是新的业务广度，而是这四件事：

1. 后端测试
2. Web 性能治理
3. Sass 技术债清理
4. 多角色真实回归

把这四件事补平之后，项目后续再扩展新能力，成本会明显低很多。
