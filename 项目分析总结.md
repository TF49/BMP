# 羽擎（BMP）项目分析总结

> 分析日期：2026-05-01  
> 分析口径：以当前仓库代码、目录、构建结果和现行文档为准  
> 本文档只负责回答三件事：项目现在处在哪个阶段、主要问题是什么、下一步优先做什么。

## 一、阶段判断

当前 BMP 已经不是“开发初期项目”，也不是“只搭出雏形的演示项目”。

更准确的判断是：

**它已经完成了核心业务铺面，进入了多端联调、复杂度治理和文档收口阶段。**

这个判断的依据不是口号，而是仓库里已经同时存在：

- 成体系的后端业务域
- 多角色 Web 入口
- UniApp 用户端、移动管理端、教练端入口
- WebSocket、定时任务、权限控制、支付退款等持续运行型能力

## 二、已经比较成熟的部分

### 1. 后端业务核心

后端是项目最成熟的一层。

- 业务域覆盖完整
- 模块组织稳定
- 权限控制不只停留在前端
- 已经具备状态流转、财务联动、推送、调度等业务系统特征

这说明项目不是“页面很多但后端空心”，而是后端中台已经成型。

### 2. Web 端主功能面

Web 端已经承担：

- 官网入口
- 管理端
- 用户端
- 教练端

从职责范围看，它已经是一个“单工程多场景”的成熟前端，而不是单一后台壳子。

### 3. UniApp 多角色入口

UniApp 不再只是普通用户补充端。

当前代码里已经能确认：

- `USER`、`MEMBER` 用户端入口
- `PRESIDENT`、`VENUE_MANAGER` 移动管理端入口
- `COACH` 教练端入口与启动分流页

这意味着移动端能力已经真正铺开。

## 三、当前真正的主要问题

### 1. 测试和验收落后于代码推进

现在最明显的问题不是“没功能”，而是：

**代码完成度高于测试完成度。**

尤其是：

- 后端自动化测试偏弱
- UniApp 多角色主链路仍依赖手工联调
- 教练端处于“代码已落地、验收仍需继续”的状态

### 2. Web 工程复杂度开始变重

Web 端一个工程承载官网、后台、用户端、教练端，短期高效，长期会带来：

- 包体积偏大
- 依赖治理成本上升
- 职责边界越来越宽

这已经不是潜在风险，而是正在出现的治理压力。

### 3. 文档曾长期滞后于代码

这轮收口前，根目录文档里同时存在：

- Android 现状残留描述
- `COACH` 移动端未开放的旧口径
- 多份文档重复讲结构、状态和判断

所以文档问题本质上不是“有没有写”，而是**写得太多但职责没切开**。

## 四、项目当前的核心矛盾

如果把项目现状浓缩成一句更实的话：

**BMP 当前的主要矛盾，不是功能不够，而是现有功能已经不少，测试、治理和文档一致性还没完全追上。**

这会直接影响三件事：

1. 后续改动时的回归成本
2. 新人理解项目边界的成本
3. 多端继续扩展时的维护成本

## 五、下一步优先级

### 第一优先级：联调与验收

重点不是再补一个新页面，而是把现有链路跑实：

- UniApp 用户端关键业务流
- 移动管理端与 Web 管理端数据一致性
- 教练端主链路
- 多角色数据隔离

### 第二优先级：后端测试补护栏

优先覆盖高业务耦合模块：

- 场地预约
- 课程预约
- 赛事报名
- 器材租借
- 财务联动

### 第三优先级：Web 包体积治理

重点看：

- Element Plus
- ECharts
- 路由级拆分
- 公共依赖加载链路

### 第四优先级：继续收敛文档

当前已经统一成三份主文档分工，后续原则应保持稳定：

- `README` 讲现状
- `项目分析总结` 讲判断
- `项目开发文档` 讲详细能力

不要再让同类信息在多份文档里并行漂移。

## 六、结论

当前 BMP 最准确的阶段定位是：

**核心业务已成型，项目进入联调、治理和维护能力建设阶段。**

如果后续继续投入，最值钱的事不是再扩一个新模块，而是把：

- 联调验收
- 后端测试
- Web 治理
- 文档一致性

这四件事补平。
