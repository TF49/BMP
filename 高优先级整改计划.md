# 高优先级整改计划

> 生成日期：2026-04-25  
> 适用项目：`D:\Developer Tool\IDEA\JAVAEE\BMP`

## 一、目标

统一当前项目的角色口径，并优先修正多端分流中的不一致问题。最终目标如下：

- `USER` 与 `MEMBER` 都归入用户端，共用同一套用户侧页面与入口
- `PRESIDENT` 与 `VENUE_MANAGER` 归入管理端，共用同一套管理端框架
- `VENUE_MANAGER` 与 `PRESIDENT` 的差异主要体现在数据权限和少量操作权限，不体现在产品形态拆分
- `COACH` 暂不纳入本轮改造，只为后续扩展预留位置

## 二、核心口径

### 1. 角色归类

- 用户端角色：`USER`、`MEMBER`
- 管理端角色：`PRESIDENT`、`VENUE_MANAGER`
- 教练端角色：`COACH`

### 2. 角色分流原则

- `USER` 和 `MEMBER` 在 Web 与 UniApp 中都进入同一套用户端
- `PRESIDENT` 和 `VENUE_MANAGER` 在 Web 与 UniApp 中都进入同一套管理端
- `VENUE_MANAGER` 与 `PRESIDENT` 的差异通过权限控制和数据范围控制实现
- `COACH` 本轮保持独立口径，不并入用户端或管理端

## 三、第一阶段高优先级整改内容

### 1. 统一角色矩阵

建立一份唯一角色规则，并同步到 Web、UniApp、文档中：

- 用户端角色：`USER`、`MEMBER`
- 管理端角色：`PRESIDENT`、`VENUE_MANAGER`
- 教练端角色：`COACH`
- 不再保留“文案允许但代码拦截”或“代码放行但文档未同步”的旧口径

需要同步到以下位置：

- Web 路由守卫与登录跳转
- UniApp 登录放行、首页分流、页面守卫
- 角色工具函数与角色文案提示
- 根目录相关说明文档

### 2. 修正 UniApp 当前最高优先级问题

围绕 `BMP-uniapp` 做第一阶段收敛：

- 修改角色判断工具，使 `MEMBER` 与 `USER` 等价进入用户端
- 将 `VENUE_MANAGER` 纳入移动管理端准入范围
- 让 `PRESIDENT` 与 `VENUE_MANAGER` 共用现有 `president` 管理端框架
- 登录成功后的分流规则统一为：
  - `USER` / `MEMBER` -> 用户端首页
  - `PRESIDENT` / `VENUE_MANAGER` -> 管理端首页
  - `COACH` -> 暂不开放，给出明确提示
- 所有“不支持当前入口”的弹窗文案同步修改，不能再把 `MEMBER` 或 `VENUE_MANAGER` 误判为不支持

### 3. 对齐 Web 端角色分流

Web 端本轮不做大改版，但要统一规则：

- 所有将 `USER` 与 `MEMBER` 区分对待的登录分流、守卫、导航判断，统一按“用户端角色”处理
- 所有将 `PRESIDENT` 与 `VENUE_MANAGER` 结构性拆开的地方，统一按“同框架，不同权限”思路处理
- 保留现有管理端能力差异，但差异应来自权限控制，不来自入口割裂

### 4. 同步后端与文档口径

后端本轮以核对和补充说明为主：

- 核对服务层数据权限是否满足：
  - `PRESIDENT` 看全量
  - `VENUE_MANAGER` 看自己场馆
  - `USER` / `MEMBER` 看自己的数据
- 如果个别接口仍按旧角色假设写死，需要补齐 `MEMBER` 作为用户侧角色的兼容
- 更新根目录文档中的角色矩阵、端能力说明、UniApp 支持范围，避免继续漂移

## 四、测试与验收

### 1. UniApp 登录分流

- `USER` 登录进入用户端
- `MEMBER` 登录进入用户端
- `PRESIDENT` 登录进入移动管理端
- `VENUE_MANAGER` 登录进入移动管理端
- `COACH` 登录保持明确提示未开放

### 2. UniApp 页面访问

- 用户侧账号不能进入管理页
- 管理侧账号能进入管理页
- `VENUE_MANAGER` 进入管理页后仅能看到自己场馆数据

### 3. Web 登录分流与路由守卫

- `USER` / `MEMBER` 始终进入用户端
- `PRESIDENT` / `VENUE_MANAGER` 始终进入管理端

### 4. 后端权限验证

- `MEMBER` 能正常访问用户侧接口
- `VENUE_MANAGER` 在核心管理接口上只能访问本场馆数据
- `PRESIDENT` 保持全量权限

### 5. 文档核对

- `README.md`
- `项目分析总结.md`
- `项目结构分析.md`

以上文档中的角色说明需与代码一致。

## 五、实施边界

本计划第一阶段只优先解决角色一致性问题，不扩做以下内容：

- 不在本轮新增教练端 UniApp 页面
- 不在本轮重构管理端整体导航
- 不在本轮新增全新角色体系
- 不在本轮做大规模 UI 改版

## 六、默认假设

- `USER` 与 `MEMBER` 业务上属于同一用户端，只是身份或等级不同
- `PRESIDENT` 与 `VENUE_MANAGER` 后续共用管理端框架，不单独再拆第二套移动管理产品
- `COACH` 本轮不开放 UniApp 端，只保留后续接入空间
- 第一阶段优先解决角色一致性，不顺手扩做教练端或重新设计移动管理端导航
