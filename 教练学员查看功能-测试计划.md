# 教练学员查看功能测试计划

## 1. 测试目标

验证教练学员查看功能 P1—P6 的正确性、安全性和可用性，重点确认 BMP-uniapp 微信小程序教练端能够实时接收以下三类 WebSocket 事件：

- `COACH_STUDENT_NEW_BOOKING`：产生新的有效课程预约。
- `COACH_STUDENT_CANCELLED`：普通用户或会员主动取消、删除本人有效预约。
- `COACH_STUDENT_ATTENDANCE_CHANGED`：教练完成签到、完成课程或登记缺席。

同时确认 Vue Web 管理端现有 `/ws` SockJS 功能不受影响，小程序在 WebSocket 断开时仍可通过 `onShow` 和下拉刷新正常使用 P1—P5 功能。

## 2. 测试范围

### 2.1 本次重点范围

- Java 后端 `/ws-native` 原生 STOMP 端点。
- STOMP CONNECT JWT 鉴权和 `/user/queue/notifications` 点对点订阅。
- 新报名、学员取消、考勤变化的事务提交后异步推送。
- BMP-uniapp 教练角色连接、重连、前后台切换和退出断开。
- 新报名角标、取消弹窗节流、200ms 页面刷新防抖。
- 教练工作台、我的学员、学员详情和课程学员名单实时刷新。
- P1—P5 查询、分页、筛选、隐私、权限和考勤操作回归。

### 2.2 不在本次范围

- `COACH_MEMBER_STATUS_CHANGED` 会员状态变化事件。
- Vue Web 页面新增教练学员实时交互。
- 短信、公众号模板消息或微信订阅消息。

## 3. 测试环境

### 3.1 软件环境

- JDK：17。
- Maven：与项目 `pom.xml` 兼容的 3.x 版本。
- Node.js：项目当前开发版本。
- 微信开发者工具：稳定版，基础库选择项目支持的最低版本及最新版各验证一次。
- 微信真机：至少一台 Android 和一台 iOS；若条件有限，至少完成一台真机验证。
- MySQL、Redis及后端依赖服务按开发环境配置启动。

### 3.2 网络与域名

- 开发者工具可使用 `ws://127.0.0.1:9090/ws-native`，并按需要关闭合法域名校验。
- 真机必须把 API 地址改为手机可访问的局域网地址，或使用已备案 HTTPS/WSS 域名。
- 生产验证必须在微信小程序后台配置合法 `request` 和 `socket` 域名。
- 后端 `WS_ALLOWED_ORIGINS` 必须包含实际使用的来源；开发环境可按现有配置使用 `*`。

## 4. 账号与数据准备

准备以下账号，账号之间不得复用同一登录会话：

| 编号 | 角色 | 用途 |
|---|---|---|
| A | `COACH` | 已绑定教练档案，接收实时事件并操作考勤 |
| B | `USER` 或 `MEMBER` | 预约和主动取消 A 教练的课程 |
| C | `PRESIDENT` | 管理员代报、退款及越权回归 |
| D | `VENUE_MANAGER` | 场馆数据范围和管理员取消语义回归 |
| E | 另一个 `COACH` | 点对点隔离和一分钟弹窗按教练账号隔离 |

准备测试数据：

1. A 教练名下至少两门课程：一门未来可报名课程，一门已开始且处于考勤窗口的课程。
2. 未来课程剩余名额至少 2 个，状态为报名中。
3. B 账号已关联有效会员档案。
4. 考勤课程至少有一条 B 的已支付预约。
5. E 教练名下准备一门独立课程，用于验证消息不会串号。
6. 记录测试前的预约 ID、课程 ID、教练 ID、会员 ID和当前报名人数。

## 6. P6 WebSocket 双账号用例

### WS-01 教练角色建立连接

前置条件：A 登录小程序，后端运行正常。

步骤：

1. 打开微信开发者工具网络面板。
2. 使用 A 登录并进入教练工作台。
3. 检查 `/ws-native` WebSocket 握手。
4. 检查 STOMP CONNECT headers 携带 `token` 和 `Authorization: Bearer <token>`。
5. 检查客户端订阅 `/user/queue/notifications`。

预期：连接成功，仅建立一个业务 STOMP 连接，无循环重连。

### WS-02 非教练角色不连接

步骤：分别使用 B、C、D 登录小程序并观察网络连接。

预期：不得建立教练 `/ws-native` 连接；切换账号后旧教练连接已关闭。

### WS-03 新报名实时刷新

步骤：

1. A 停留在教练工作台，记录“我的学员”角标和学员总数。
2. B 预约 A 的未来课程并确认预约成功。
3. 观察 A 的提示和页面变化。
4. A 点击“我的学员”进入列表。

预期：

- A 收到 `COACH_STUDENT_NEW_BOOKING`。
- “我的学员”入口角标加 1，进入列表后角标清零。
- 当前相关页面在约 200ms 合并窗口后刷新。
- 学员列表包含 B，不显示取消弹窗，不触发震动。
- E 教练不收到该消息。

### WS-04 学员主动取消实时提示

步骤：

1. A 停留在学员列表或课程学员名单。
2. B 主动取消本人有效预约。
3. 30 秒内让 B 再取消另一条预约。
4. 距第一次提示超过 60 秒后，再产生一次 B 的主动取消。

预期：

- 每次取消均触发页面数据刷新。
- 第一次取消短震动并弹出“学员取消预约”。
- 同一教练一分钟内第二次取消不重复弹窗，但数据仍刷新。
- 超过一分钟后可再次弹窗。

### WS-05 取消弹窗按教练隔离

步骤：在一分钟内分别让 A、E 各收到一次学员主动取消。

预期：A 和 E 均能看到各自第一次取消弹窗，节流状态不得跨教练账号共享。

### WS-06 教练考勤变化静默刷新

步骤：

1. A 在课程学员名单对 B 依次执行签到、完成或登记缺席中的合法操作。
2. 同时打开 B 的学员详情或重新进入详情页观察记录。
3. 重复提交当前已经生效的同一个考勤命令。

预期：

- 实际状态更新后收到 `COACH_STUDENT_ATTENDANCE_CHANGED`。
- 页面刷新，不弹窗、不震动。
- `bookingStatus` 和 `attendanceStatus` 与数据库一致。
- 幂等重复命令不产生重复事件。

### WS-07 教练取消不冒充学员取消

步骤：A 在课程开始前通过教练端取消一条预约。

预期：业务取消成功，但 A 不收到 `COACH_STUDENT_CANCELLED` 弹窗。

### WS-08 系统取消和管理员退款不冒充学员取消

步骤：

1. 制造一条超时未支付预约并执行自动取消。
2. 使用 C 对已支付预约执行退款。

预期：不得向教练发送 `COACH_STUDENT_CANCELLED`；原有订单状态通知按现有逻辑执行。

### WS-09 事务回滚不推送

步骤：制造新增预约后课程报名人数更新失败，或在测试环境让事务最终回滚。

预期：数据库无有效新增记录，A 不收到新报名事件。

### WS-10 推送故障不影响主业务

步骤：临时断开 WebSocket broker发送链路或让异步任务拒绝，再执行报名、取消或考勤。

预期：主业务按数据库事务结果成功或失败，不因 WebSocket 推送异常回滚已成功业务；后端记录警告日志。

## 7. 生命周期与断线恢复

### LIFE-01 前后台切换

1. A 登录并确认连接成功。
2. 将小程序切到后台。
3. 在后台期间刷新 token 或等待一段时间。
4. 返回前台。

预期：后台连接关闭且不会因 token 变化偷偷重连；返回前台后只恢复一个最新 token 连接。

### LIFE-02 退出登录

步骤：A 退出登录。

预期：连接立即断开，停止重连；旧账号角标不会显示给后续非教练账号。

### LIFE-03 token 刷新竞态

步骤：在网络较慢时触发 token 刷新并快速前后台切换。

预期：最终只保留最新 token 对应连接，旧连接不得覆盖新连接。

### LIFE-04 网络中断重连

1. A 保持前台，关闭网络 10 秒以上。
2. 恢复网络。

预期：客户端按约 3 秒间隔尝试恢复；恢复后重新订阅且不重复处理单条消息。

### LIFE-05 WebSocket 不可用时的刷新兜底

步骤：阻断 `/ws-native`，分别进入教练工作台、学员列表、学员详情和课程学员名单，并执行下拉刷新。

预期：页面仍通过 HTTP 正常加载；`onShow` 和下拉刷新均可用，不因 WebSocket 失败阻断页面。

## 8. P1—P5 回归

### 8.1 我的学员列表

- 默认分页、上一页/下一页正确。
- 关键词、风险、今日有课、会员类型筛选正确。
- 最近上课、出勤率、消费和课时排序正确。
- 加载中、空状态、失败重试和下拉刷新正确。

### 8.2 学员详情

- 概览、训练进度、课程安排、出勤记录和消费记录加载正确。
- 课程安排按待上课与历史记录分组。
- 月度出勤率只统计完成和缺席。
- 风险提示和会员到期提示符合数据。

### 8.3 教练考勤

- 课程开始前禁止考勤。
- 签到、完成、缺席及 24 小时纠错窗口正确。
- 课程结束 24 小时后禁止更正。
- 并发状态变化返回冲突提示，不覆盖他人操作。

### 8.4 缓存和刷新

- 新报名、取消和考勤后相关学员查询缓存失效。
- 实时刷新与手动刷新结果一致。
- 快速连续事件只合并页面刷新，不丢失取消事件业务数据。

## 9. 权限与隐私

### 9.1 权限

- A 只能查看与本人课程相关的学员。
- A 访问 E 的学员详情返回拒绝或不存在。
- B、C、D 不能通过教练学员接口越权读取数据。
- 多角色 `VENUE_MANAGER + MEMBER` 的管理员操作不得发送“学员主动取消”事件。
- STOMP 使用无 token、过期 token、伪造 token 连接均应失败。

### 9.2 隐私

抓取三类 WebSocket 消息并确认载荷中不存在：

- 手机号。
- 身份证号。
- 余额。
- 累计充值金额。
- 无关消费明细。

页面验证：列表手机号保持脱敏；详情仅在已有授权交互中按现有规则展示；WebSocket 不得扩大隐私数据范围。

## 10. Vue Web 兼容回归

1. 启动现有 Vue Web 管理端。
2. 确认仍通过 `/ws` SockJS 连接。
3. 验证原有订单状态、管理待办和看板刷新功能。

预期：`/ws` 未被删除，SockJS 行为不变；新增 `/ws-native` 不影响 Web 页面。

## 11. 真机专项检查

- Android、iOS 各验证一次登录、前后台、断网重连和取消震动。
- 确认生产使用 `wss`，证书链有效且域名已加入微信后台白名单。
- 确认真机运行时支持 STOMP 使用的 UTF-8 编解码能力；若出现 `TextEncoder` 或 `TextDecoder` 不存在，记录机型、系统和微信基础库版本并补兼容垫片。
- 检查弱网下无重复弹窗、无连接泄漏、无持续高频重连。
- 检查开发者工具和真机控制台无未捕获异常。

## 12. 验收标准

满足以下条件方可验收：

1. 后端和小程序自动化命令全部通过。

3. 事务回滚不推送，推送故障不影响主业务。

5. 取消弹窗一分钟节流、200ms 刷新防抖和新报名角标正确。
6. 前后台、退出登录、token 刷新和断线重连无旧连接覆盖。
7. P1—P5 回归、权限和隐私检查无阻断问题。
8. 微信真机至少完成一轮核心流程。
