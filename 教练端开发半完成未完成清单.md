# UniApp 教练端开发完成情况清单

更新时间：2026-04-30  
范围：`BMP-uniapp` 教练端（`pages/coach/**`）及其相关登录分流、接口接入、联调测试

## 一、当前总体判断

当前 UniApp 教练端已经从“原型可访问”推进到“P0 主链路已落地，P1 首批扩展页已接入真实数据”的状态。

已完成的核心能力：

- `COACH` 角色已放开移动端准入，不再被登录页直接拦截
- 已新增教练端启动分流页，登录后会先校验 `/api/coach/me`
- 已绑定账号可进入教练工作台，未绑定账号可进入专门提示页
- 教练工作台、我的课程、预约明细、我的档案均已接入真实接口
- 我的档案已支持保存 `coachName / phone / specialty / experience / avatar`
- 预约明细已支持查看详情，以及签到、完成、缺席/取消状态更新
- 课表页已接真实日视图
- 学员名单页已接真实预约数据，并依赖课程上下文进入
- 教练端底部导航已改为 `reLaunch`，避免页面栈不断累积

当前仍未达到“全部功能完成”的原因主要是：

- 教练端主链路尚未完成真机或开发环境的完整手工联调
- 模块 21 还没有完成正式勾验，只同步到“代码已实现，待联调”

当前更准确的定义应为：

**“UniApp 教练端 P0 主链路已完成，P1 的课表日视图、课程上下文学员名单和课程历史页都已落地，可进入联调与试用验证阶段；剩余工作主要集中在联调验收和页面职责持续收敛。”**

## 四、当前仍需注意的细节

### 1. `schedule / students / history` 已全部进入真实接口阶段

这三页都已经不再是演示态或占位页。  
但“代码已落地”不等于“联调已通过”，仍需要后续用真实教练账号把链路跑顺。

### 2. 现有自动化验证只覆盖类型检查与现有单测

当前已完成的验证包括：

- `npm.cmd run type-check`
- `npm.cmd test`

这说明当前代码没有明显类型错误，且现有单测未回归。  
但它们不能替代真实教练账号在开发环境或真机上的流程联调。

### 3. 请求失败提示策略已经收敛，但仍需联调确认

当前教练本人接口已通过 `suppressErrorToast` 避免大部分重复 toast。  
仍建议在联调时重点看未绑定、弱网、401 过期三类场景，确认不会出现双提示或错误停留页。
