> **文档版本**: v2.3.0

> **数据库版本**: MySQL 8.0.28
> **字符集**: utf8mb4_unicode_ci

## 一、设计概述

根据开发文档中的功能需求分析，已设计并实现 22 个核心业务表，覆盖场馆管理、会员管理、场地管理、器材管理、教练管理、课程管理、预约管理、赛事管理、财务管理（含审计与对账）、搜索服务、活动管理等所有业务模块。**初始化脚本**：仅使用 `src/main/resources/sql/badminton.sql` 即可完成全库建表与示例数据初始化，无需单独执行其他 SQL 文件。

## 二、表结构设计

### 2.1 核心业务表（按优先级排序）

#### 0. sys_user（用户表）🔴 最高优先级

- **用途**：管理系统用户信息，支持三角色权限体系
- **字段设计**：
  - `id` BIGINT 主键自增
  - `username` VARCHAR(50) 用户名（唯一）
  - `password` VARCHAR(100) 密码
  - `phone` VARCHAR(20) 手机号
  - `id_card` VARCHAR(18) 身份证号
  - `avatar` VARCHAR(255) 头像URL
  - `role` VARCHAR(20) 角色（PRESIDENT-协会会长，VENUE_MANAGER-场馆管理者，USER-普通用户）
  - `venue_id` BIGINT 场馆ID（外键关联sys_venue，仅场馆管理者需要，协会会长和普通用户为空）
  - `status` TINYINT(1) 状态（0-禁用，1-启用）
  - `create_time` DATETIME 创建时间
  - `update_time` DATETIME 更新时间
  - `last_login_time` DATETIME 最后登录时间
- **索引**：`idx_username`, `idx_id_card`, `idx_role`, `idx_venue_id`
- **外键**：`venue_id` REFERENCES `sys_venue(id)`
- **重要约束**：
  - 协会会长（PRESIDENT）只能存在一个（应用层约束）
  - 场馆管理者（VENUE_MANAGER）必须关联一个场馆（venue_id不能为空）
  - 普通用户（USER）的venue_id必须为空

#### 1. sys_venue（场馆表）🔴 最高优先级

- **用途**：管理羽毛球场馆信息
- **字段设计**：
  - `id` BIGINT 主键自增
  - `venue_name` VARCHAR(100) 场馆名称（必填）
  - `address` VARCHAR(255) 地址
  - `contact_phone` VARCHAR(20) 联系电话
  - `contact_person` VARCHAR(50) 联系人
  - `business_hours` VARCHAR(100) 营业时间（如：06:00-24:00）
  - `description` TEXT 场馆描述
  - `venue_image` VARCHAR(255) 场馆图片URL（主图，保留字段，向后兼容）
  - `latitude` DECIMAL(10,7) 纬度（预留，暂不使用）
  - `longitude` DECIMAL(10,7) 经度（预留，暂不使用）
  - `status` TINYINT(1) 状态（0-关闭，1-营业中，2-暂停）
  - `create_time` DATETIME 创建时间
  - `update_time` DATETIME 更新时间
- **索引**：`idx_venue_name`, `idx_status`

#### 1.1 sys_venue_image（场馆图片表）🟡 中优先级

- **用途**：管理场馆的多张图片，支持主图和详情图

#### 1.3 sys_venue_status_log（场馆状态变更日志表）🟡 中优先级

#### 2. sys_member（会员表）🔴 最高优先级

- **用途**：管理会员信息
- **字段设计**：
  - `id` BIGINT 主键自增
  - `user_id` BIGINT 用户ID（外键关联sys_user，可为空）
  - `member_name` VARCHAR(50) 姓名（必填）

  - `phone` VARCHAR(20) 电话
  - `id_card` VARCHAR(18) 身份证号
  - `member_type` VARCHAR(20) 会员类型（NORMAL-普通用户，MEMBER-会员，默认NORMAL）
  - `member_level` INT 会员等级（1-5级，用于会员等级划分，默认1）
  - `register_time` DATETIME 注册时间
  - `expire_time` DATETIME 到期时间（仅会员MEMBER需要，普通用户NORMAL可为空）
  - `status` TINYINT(1) 状态（0-冻结，1-正常，2-到期，默认1）
  - `total_consumption` DECIMAL(10,2) 累计消费金额（默认0.00）
  - `balance` DECIMAL(10,2) 账户余额（默认0.00）
  - `create_time` DATETIME 创建时间
  - `update_time` DATETIME 更新时间
  - `del_flag` TINYINT(1) 逻辑删除标记（0-正常，1-删除，默认0）
- **索引**：`idx_user_id`, `idx_phone`, `idx_id_card`, `idx_member_type`, `idx_status`
- **外键**：`user_id` REFERENCES `sys_user(id) ON DELETE SET NULL`

#### 2.x biz_recharge_record（充值记录表）🔴 最高优先级

- **用途**：记录会员充值记录，支持用户自己充值和管理员充值两种方式
- **字段设计**：
  - `id` BIGINT 主键自增
  - `member_id` BIGINT 会员ID（外键关联sys_member）
  - `user_id` BIGINT 操作用户ID（外键关联sys_user，可为空，用户自己充值时记录）
  - `recharge_amount` DECIMAL(10,2) 充值金额
  - `recharge_method` VARCHAR(20) 充值方式（CASH-现金，ALIPAY-支付宝，WECHAT-微信，BANK-银行转账）
  - `recharge_time` DATETIME 充值时间
  - `operator_type` VARCHAR(20) 操作类型（USER-用户自己充值，ADMIN-管理员充值）
  - `operator_id` BIGINT 操作人ID（管理员充值时记录管理员ID）
  - `remark` VARCHAR(500) 备注
  - `is_upgraded` TINYINT(1) 是否触发升级（0-否，1-是）
  - `venue_id` BIGINT 所属场馆ID（操作人或业务关联，便于场馆级查询）
  - `create_time` DATETIME 创建时间
- **索引**：`idx_member_id`, `idx_user_id`, `idx_venue_id`, `idx_recharge_time`, `idx_is_upgraded`
- **外键**：`member_id` REFERENCES `sys_member(id) ON DELETE CASCADE`, `user_id` REFERENCES `sys_user(id) ON DELETE SET NULL`, `venue_id` REFERENCES `sys_venue(id) ON DELETE SET NULL`

#### 2.y biz_member_consume_record（会员消费记录表）🔴 最高优先级

- **用途**：记录会员所有业务消费与退款（退款使用负数金额冲正），用于会员管理模块展示“消费记录”和对账。
- **业务来源覆盖**：场地预约（BOOKING）、课程预约（COURSE）、器材租借（EQUIPMENT）、赛事报名（TOURNAMENT）、其他（OTHER）。
- **字段设计**：
  - `id` BIGINT 主键自增
  - `member_id` BIGINT 会员ID（外键关联sys_member）
  - `business_type` VARCHAR(50) 业务类型（BOOKING/COURSE/EQUIPMENT/TOURNAMENT/OTHER）
  - `business_id` BIGINT 业务ID（关联对应业务表的ID，可为空）
  - `amount` DECIMAL(10,2) 金额（正数-消费，负数-退款/冲正）
  - `payment_method` VARCHAR(20) 支付方式（沿用业务表定义，含BALANCE）
  - `payment_status` TINYINT(1) 支付状态（0-未支付，1-已支付，2-已退款，默认0）
  - `before_balance` DECIMAL(10,2) 扣款前余额（仅余额相关时记录）
  - `venue_id` BIGINT 所属场馆ID（业务所属场馆，便于场馆级查询）
  - `after_balance` DECIMAL(10,2) 扣款后余额（仅余额相关时记录）
  - `remark` VARCHAR(500) 备注
  - `create_time` DATETIME 创建时间
- **索引**：`idx_member_id`, `idx_venue_id`, `idx_business (business_type, business_id)`, `idx_payment_status`, `idx_create_time`
- **外键**：`member_id` REFERENCES `sys_member(id) ON DELETE CASCADE`, `venue_id` REFERENCES `sys_venue(id) ON DELETE SET NULL`

## 四、设计规范遵循

### 4.1 命名规范

- ✅ 表名使用小写字母和下划线（如：`sys_venue`）
- ✅ 字段名使用小写字母和下划线（如：`venue_name`）
- ✅ 主键统一使用`id`
- ✅ 创建时间：`create_time`
- ✅ 更新时间：`update_time`
- ✅ 逻辑删除：`del_flag`（0-正常，1-删除）

#### 4.1.1 API 字段命名约定（与后端约定）

为保证前后端一致性，文档约定：
- API 请求与响应使用 camelCase（示例：`venueId`、`rechargeAmount`）
- 数据库字段使用 snake_case（示例：`venue_id`、`recharge_amount`）
- 后端负责在 DTO/实体层做 camelCase <-> snake_case 的映射，接口示例均使用 camelCase


### 4.2 数据类型规范

- 主键：`BIGINT AUTO_INCREMENT`
- 字符串：`VARCHAR`（根据实际长度）
- 文本：`TEXT`
- 金额：`DECIMAL(10,2)`
- 状态：`TINYINT(1)`
- 时间：`DATETIME`（日期时间）、`DATE`（日期）、`TIME`（时间）

### 4.3 索引设计

- 主键自动创建聚簇索引
- 外键字段创建索引
- 常用查询字段创建索引（如：状态、日期、编号等）
- 唯一约束创建唯一索引

## 七、注意事项

1. **用户角色约束**：
   - 协会会长（PRESIDENT）只能存在一个，需要在应用层实现唯一性检查
   - 场馆管理者（VENUE_MANAGER）必须关联一个场馆（venue_id不能为空）
   - 普通用户（USER）的venue_id必须为空
2. **外键约束**：
   - 使用 `ON DELETE CASCADE`：当父表记录删除时，子表相关记录自动删除（适用于强关联关系，如场馆图片、营业时间、场地、器材、教练等）
   - 使用 `ON DELETE SET NULL`：当父表记录删除时，子表外键字段设置为NULL（适用于可选关联关系，如会员关联用户等）
   - 建议在应用层也进行数据完整性控制，双重保障
2. **逻辑删除**：所有业务表都使用`del_flag`字段实现逻辑删除，不进行物理删除（场馆表除外）
3. **唯一约束**：
   - 业务单号（预约单号、租借单号、财务单号等）需要设置唯一约束
   - 器材编号需要全局唯一
   - 场地编号在同一场馆内唯一（联合唯一索引：venue_id + court_code）
4. **场馆关联**：
   - 场地（sys_court）、器材（sys_equipment）、教练（sys_coach）都必须关联场馆（venue_id必填）
   - 不同场馆拥有独立的场地、器材和教练资源
   - 删除场馆时，相关场地、器材、教练会自动级联删除
4. **业务单号自动生成**：使用数据库触发器或函数自动生成业务单号
   - 预约单号：BK+日期+序号（如：BK20250118001）
   - 课程预约单号：CB+日期+序号（如：CB20250118001）
   - 器材租借单号：ER+日期+序号（如：ER20250118001）
   - 赛事报名单号：TR+日期+序号（如：TR20250118001）
   - 财务单号：FN+日期+序号（如：FN20250118001）
5. **字符集**：统一使用`utf8mb4`字符集，支持中文和emoji
6. **时间字段**：`create_time`使用`DEFAULT CURRENT_TIMESTAMP`，`update_time`在应用层更新
7. **金额字段**：统一使用`DECIMAL(10,2)`，确保精度
8. **状态字段**：统一使用`TINYINT(1)`，并在注释中说明状态值含义
9. **会员到期时间**：`expire_time`字段仅会员（MEMBER）类型需要，普通用户（NORMAL）可为空
10. **场地计费方式**：通过`billing_type`字段区分按小时计费（HOUR）或按次计费（TIME）

## 八、后续优化建议

1. **分表策略**：如果数据量较大，可以考虑按时间分表（如财务表按月分表）
2. **读写分离**：为统计分析模块准备，可以考虑读写分离
3. **缓存策略**：场馆、场地等基础数据可以加入Redis缓存
4. **审计日志**：已实现财务审计日志表（`biz_finance_audit_log`）及财务对账记录表（`biz_finance_reconciliation`），记录财务操作与对账结果；其他模块可按需扩展操作日志表
5. **配置表**：可以考虑添加系统配置表管理预约规则、价格策略等
