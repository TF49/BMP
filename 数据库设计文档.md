# 数据库设计文档

> **文档版本**：v3.6.0
> **项目版本**：1.0.0（功能开发已完成）  
> **最后更新**：2026-07-16
> **数据库版本**：MySQL 8.x  
> **字符集**：`utf8mb4_unicode_ci`  
> **文档定位**：说明 BMP 数据库范围、表目录、关键关系、业务约束与初始化方式。

## 1. 设计概览

### 1.1 当前事实来源

当前数据库结构以以下脚本为唯一事实来源：

```text
src/main/resources/sql/badminton.sql
```

这个脚本当前已经包含：

- 建库、建表、索引、外键
- 初始化数据
- 预约主单/明细模型
- 三套场地价格字段
- 课程预约考勤字段、教练学员查询索引与历史完成数据回填
- 财务审计、财务对账、用户设置、反馈、取消政策等增量表

### 1.2 当前表规模

基于 2026-07-16 对 `badminton.sql` 中 `CREATE TABLE` 的复核，当前库内共有 **27 张物理表**。

与旧版文档相比，需要特别纠正两点：

1. 当前库里 **没有** 独立的 `biz_activity` 表，近期动态由接口层聚合生成  
2. 当前库里 **已经有** `biz_cancel_policy`、`biz_feedback`、`sys_user_setting` 等此前未完整同步到文档的表
3. 当前库里 **没有** 独立教练学员关系表；关系通过课程与课程预约实时推导

## 3. 初始化方式

### 3.1 新库初始化

推荐直接执行：

```text
src/main/resources/sql/badminton.sql
```

当前不需要再拼接执行其它拆分 SQL 文件。

### 3.2 初始化后建议检查

建议至少确认以下项目：

```sql
SHOW TABLES;
SELECT id, username, role FROM sys_user WHERE username = 'admin';
```

演示账号（密码均为 **`123456`**，BCrypt 存储）：`admin`（`PRESIDENT`）、`venue_manager_1`（`VENUE_MANAGER`）、`user001`（`USER`）。公开发布仓库前请按需脱敏或删除个人数据。

并确认以下关键表存在：

- `biz_booking`
- `biz_booking_court`
- `biz_finance`
- `biz_finance_audit_log`
- `biz_finance_reconciliation`
- `biz_cancel_policy`
- `biz_feedback`
- `sys_user_setting`

并确认考勤字段和查询索引已生效：

```sql
SHOW COLUMNS FROM biz_course_booking LIKE 'attendance_status';
SHOW COLUMNS FROM biz_course_booking LIKE 'actual_checkin_time';
SHOW COLUMNS FROM biz_course_booking LIKE 'actual_finish_time';
SHOW INDEX FROM biz_course_booking WHERE Key_name LIKE 'idx_cb_%';
SHOW INDEX FROM biz_course WHERE Key_name = 'idx_c_coach_date_range';
```

### 3.3 老库升级

`badminton.sql` 含 `DROP TABLE IF EXISTS`，**不能直接用于有业务数据的老库升级**。仓库当前没有 Flyway/Liquibase 或独立增量 migration；老库需要先备份，再从全量脚本提取并改造成幂等 `ALTER TABLE`。

至少需要核对：

- `biz_course_booking.attendance_status`、`actual_checkin_time`、`actual_finish_time`
- `idx_cb_course_member_status`、`idx_cb_member_attendance`、`idx_cb_course_attendance`
- `biz_course.idx_c_coach_date_range`
- 支付超时扫描索引（如 `idx_booking_unpaid_scan`、`idx_rental_unpaid_scan`、`idx_registration_unpaid_scan`）
- 财务、场地明细和常用筛选复合索引

`src/main/resources/sql/explain-coach-students.sql` 只用于 `EXPLAIN` 教练学员查询计划，不是迁移脚本。

## 4. 关键表说明

字段最终定义以 `badminton.sql` 为准，本章只说明职责和关键语义。

### 4.1 `sys_user`

用途：

- 系统登录用户
- 角色承载主体
- 场馆管理者与教练账号来源

关键字段：

- `username`
- `password`
- `role`
- `venue_id`
- `status`

约束要点：

- `PRESIDENT` 应在应用层保持唯一
- `VENUE_MANAGER` 必须绑定场馆
- `COACH` 账号通过 `sys_coach.user_id` 与教练档案关联

### 4.2 `sys_member`

用途：

- 用户侧消费主体
- 余额、会员等级、充值统计的承载主体

关键字段：

- `user_id`
- `member_name`
- `member_type`
- `member_level`
- `balance`
- `total_recharge`
- `status`

### 4.3 `sys_user_setting`

用途：

从初始化数据可确认，当前至少覆盖：

- `BOOKING` 类型取消政策

### 4.8 `biz_course` / `biz_course_booking`

用途：

- `biz_course`：课程主表
- `biz_course_booking`：课程预约订单

特点：

- `biz_course` 通过 `coach_id`、`court_id` 关联教练和场地
- `biz_course_booking` 承载课程支付、状态流转和教练端预约视图
- 教练学员关系由 `biz_course.coach_id` 与 `biz_course_booking.member_id` 推导，不新增关系表

考勤字段：

| 字段 | 说明 |
|------|------|
| `attendance_status` | `0` 未登记、`1` 已签到、`2` 已完成、`3` 缺席 |
| `actual_checkin_time` | 实际签到时间 |
| `actual_finish_time` | 实际完成时间 |

相关索引：

- `idx_cb_course_member_status(course_id, member_id, status, attendance_status, del_flag)`
- `idx_cb_member_attendance(member_id, attendance_status, status, del_flag, course_id)`
- `idx_cb_course_attendance(course_id, attendance_status, status, del_flag)`
- `idx_c_coach_date_range(coach_id, course_date, start_time, end_time, status, del_flag)`

全量脚本先插入历史课程预约，再通过 `ALTER TABLE` 增加考勤列，并将未删除且 `status=4` 的历史完成预约回填为 `attendance_status=2`。

## 6. 当前业务约束摘要

### 6.1 预约约束

当前预约模型的数据库承载规则如下：

1. `SHARED` 仅允许关联 1 块场地
2. `PACKAGE` 允许关联多块场地，但不能覆盖整个场馆全部场地
3. 同场地同时间段下：
   - 多条 `SHARED` 可以并存
   - 存在 `PACKAGE` 时，不允许再创建 `SHARED`
   - 存在任意有效预约时，不允许该场地再加入新的 `PACKAGE`
4. `biz_booking.court_id` 当前只保留首场地兼容语义，不再作为冲突判断事实来源

### 6.2 资源归属约束

以下资源必须归属场馆：

- 场地
- 器材
- 教练
- 赛事
- 多数财务记录

### 6.3 单号前缀约束

| 业务 | 单号前缀 |
|------|------|
| 场地预约 | `BK` |
| 课程预约 | `CB` |
| 器材租借 | `ER` |
| 赛事报名 | `TR` |
| 穿线服务 | `ST` |
| 财务流水 | `FN` |
| 财务对账 | `RC` |

### 6.4 反馈与设置约束

- `biz_feedback` 同时服务联系留言与反馈，不需要拆成两张表
- `sys_user_setting` 采用 `key/value` 结构，适合继续承载轻量个性化配置

### 6.5 教练学员与考勤约束

1. 教练学员关系以课程预约为事实，不保存冗余关系记录。
2. 当前查询仅纳入有效课程，以及已支付/进行中/已完成预约或已登记考勤的预约。
3. 关系查看窗口保留到课程结束后 180 天；这是服务层常量，不是数据库字段。
4. 考勤状态机由服务层执行：未登记可签到或记缺席，已签到可完成或改缺席；数据库默认值为未登记。
5. 学员风险阈值（出勤率 70%、不活跃 14 天）来自应用配置，不写入数据库。

## 7. 当前边界与注意事项

### 7.1 当前没有独立活动表

近期动态接口 `/api/activity/recent` 当前直接聚合：

- 最近会员注册
- 最近预约
- 最近财务流水

所以数据库侧不应再引用并不存在的 `biz_activity`。

### 7.2 当前迁移策略偏“全量脚本事实源”

仓库当前更依赖 `badminton.sql` 作为总脚本，而不是维护独立的增量 migration 目录。  
这意味着：

- 新环境初始化更直接
- 老库升级必须先备份并人工生成幂等迁移，不能重跑含 `DROP TABLE` 的全量脚本
- `explain-coach-students.sql` 是查询计划检查工具，不承担结构升级

## 8. 后续优化建议

1. 为高频统计与大表查询准备归档 / 分表评估（详见 8.1）
2. 为 `biz_finance`、`biz_booking_court` 这类高活跃表继续审视索引策略（性能复合索引已合并进 `badminton.sql`）
3. 如果后续设置项继续增长，可考虑为 `sys_user_setting` 补充更明确的命名规范与缓存层
4. 若未来需要更强审计能力，可在财务之外扩展统一业务审计表

### 8.1 历史数据归档与读写分离规划（L 级用户量）

当业务规模达到 **50k+ 用户 / 500k+ 订单** 时，建议按以下步骤演进，避免单表无限增长拖慢查询与备份。

**归档目标表（优先级由高到低）**

| 表名 | 归档触发条件 | 归档策略 |
|------|-------------|----------|
| `biz_finance` | 行数 > 200 万或超过 2 个完整财年 | 按 `create_time` 年度迁入 `biz_finance_archive_YYYY` |
| `biz_booking` / `biz_booking_court` | 已完成且超过 18 个月 | 主单+明细成对迁入 `biz_booking_archive_YYYY` |
| `biz_member_consume_record` | 行数 > 300 万 | 按年度归档，保留会员近期 12 个月在主表 |
| `biz_finance_audit_log` | 行数 > 500 万 | 按 `operation_time` 归档，主表保留 12 个月 |

**归档流程（建议）**

1. 创建与主表结构一致的 `_archive_YYYY` 表（无业务写入，只读查询）。
2. 低峰期批量 `INSERT INTO archive SELECT ... WHERE create_time < cutoff LIMIT N` 循环迁移。
3. 校验归档行数与 checksum 后，从主表删除已归档数据（或保留逻辑删除标记）。
4. Dashboard / 报表接口增加「是否查归档」参数，默认仅查主表。

**读写分离**

- 主库：预约下单、支付、余额扣减等写操作。
- 从库：财务列表、会员统计、Dashboard 聚合等只读查询。
- Spring 可通过 `@Transactional(readOnly = true)` + 路由数据源（如 ShardingSphere / 自定义 AbstractRoutingDataSource）实现。

**分库分表（XL 级，可选）**

- 按 `venue_id` 水平分片业务流水表，仅在单场馆订单量极大时考虑。
- 搜索类需求（会员名、单号模糊查）可引入 Elasticsearch，MySQL 仅保留精确查询与事务。

**监控与回滚**

- 开启 MySQL `slow_query_log`，阈值建议 1s。
- 归档任务需可暂停、可回滚（归档表数据保留至下一财年审计结束）。
