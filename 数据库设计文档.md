# 数据库设计文档

> **文档版本**：v3.1.0  
> **最后更新**：2026-05-05  
> **数据库版本**：MySQL 8.x  
> **字符集**：`utf8mb4_unicode_ci`  
> **文档定位**：本手册用于说明 BMP 的数据库范围、表目录、关系结构、设计规范与初始化方式。

## 1. 设计概览

### 1.1 当前数据库范围

BMP 当前数据库已经覆盖场馆经营核心业务，包含：

- 用户、角色、会员
- 场馆、场地、器材、教练
- 预约、课程、赛事、租借、穿线
- 充值、消费、财务、审计、对账
- 通知、活动等扩展业务

当前初始化脚本以 `src/main/resources/sql/badminton.sql` 为唯一事实来源。

当前预约模型已经升级为：

- `biz_booking`：预约主单
- `biz_booking_court`：预约场地明细

当前场地价格模型已经升级为三套价格：

- `package_price_per_hour`
- `shared_price_per_hour`
- `shared_price_per_time`

### 1.2 当前表规模

当前库内共 **23 张核心业务表**。

可按职能划分为 5 组：

1. **身份与基础主数据**
2. **经营资源数据**
3. **交易与订单数据**
4. **财务与审计数据**
5. **辅助业务数据**

## 2. 数据库初始化

### 2.1 初始化方式

推荐直接执行：

```text
src/main/resources/sql/badminton.sql
```

该脚本负责：

- 建库
- 建表
- 建索引
- 建外键
- 插入示例数据

无需再额外执行其他拆分 SQL 文件。

说明：

- 当前 `badminton.sql` 已包含预约重构后的最终结构
- 新库初始化时，直接执行该文件即可
- 老库升级时，应使用与当前结构一致的增量迁移脚本思路，将旧预约迁移为“1 条主单 + 1 条明细”

### 2.2 初始化验证

建议初始化后至少检查：

```sql
SHOW TABLES;
SELECT id, username, role FROM sys_user WHERE username = 'admin';
```

同时确认以下关键表存在：

- `biz_finance`
- `biz_finance_audit_log`
- `biz_finance_reconciliation`
- `biz_stringing_service`
- `sys_notification`
- `biz_activity`

## 4. 关键表说明

本章不逐字段展开全部细节，而是说明每类关键表的职责、核心字段和主要约束。字段最终定义以 `badminton.sql` 为准。

### 4.1 用户与会员体系

#### `sys_user`

用途：

- 系统登录用户
- 角色承载主体
- 场馆管理者与教练账号来源

核心字段：

- `username`
- `password`
- `role`
- `venue_id`
- `status`

关键约束：

- `PRESIDENT` 应保持唯一
- `VENUE_MANAGER` 必须绑定 `venue_id`
- `COACH` 用于教练本人工作流入口

#### `sys_member`

用途：

- 会员身份与消费主体
- 用户端交易关联主体

核心字段：

- `user_id`
- `member_name`
- `member_type`
- `member_level`
- `balance`
- `status`

关键约束：

- 支持 `NORMAL` 与 `MEMBER`
- 允许与 `sys_user` 关联，也允许独立存在

#### `biz_recharge_record`

用途：

- 记录自助充值与管理员代充

核心字段：

- `member_id`
- `user_id`
- `recharge_amount`
- `recharge_method`
- `operator_type`
- `is_upgraded`

#### `biz_member_consume_record`

用途：

- 统一记录会员消费与退款冲正

核心字段：

- `member_id`
- `business_type`
- `business_id`
- `amount`
- `payment_method`
- `payment_status`

## 5. 关系结构

### 5.1 核心关系

```text
sys_venue
 ├─ sys_venue_image
 ├─ sys_venue_schedule
 ├─ sys_venue_status_log
 ├─ sys_court
 ├─ sys_equipment
 │   └─ sys_equipment_image
 ├─ sys_coach
 │   └─ biz_course
 │       └─ biz_course_booking
 └─ biz_tournament
     └─ biz_tournament_registration

sys_member
 ├─ biz_recharge_record
 ├─ biz_member_consume_record
 ├─ biz_booking
 │   └─ biz_booking_court
 ├─ biz_course_booking
 ├─ biz_equipment_rental
 ├─ biz_tournament_registration
 └─ biz_stringing_service

biz_finance
 ├─ biz_finance_audit_log
 └─ biz_finance_reconciliation
```

### 5.2 外键设计原则

- 强归属关系：优先使用 `ON DELETE CASCADE`
- 可选关联关系：使用 `ON DELETE SET NULL`
- 应用层仍需保留业务完整性校验，不完全依赖数据库外键

## 6. 命名与字段规范

### 6.1 命名规范

- 表名：小写 + 下划线
- 字段名：小写 + 下划线
- 主键：统一为 `id`
- 创建时间：`create_time`
- 更新时间：`update_time`
- 逻辑删除：`del_flag`

### 6.2 数据类型规范

| 类型场景 | 推荐类型 |
|------|------|
| 主键 | `BIGINT` |
| 短文本 | `VARCHAR` |
| 长文本 | `TEXT` |
| 金额 | `DECIMAL(10,2)` |
| 状态 | `TINYINT(1)` |
| 日期时间 | `DATETIME` |
| 日期 | `DATE` |
| 时间 | `TIME` |

### 6.3 索引规范

- 主键自动索引
- 外键字段必须建索引
- 常用筛选字段建普通索引
- 单号类字段建唯一索引
- 联合唯一场景使用复合索引

## 7. 业务约束摘要

### 7.1 角色与归属约束

- `PRESIDENT` 需要在应用层保持唯一
- `VENUE_MANAGER` 必须关联场馆
- `COACH` 账号通过 `sys_coach.user_id` 绑定

### 7.2 场馆归属约束

以下资源必须归属场馆：

- 场地
- 器材
- 教练
- 赛事

### 7.3 单号约束

| 业务 | 单号前缀 |
|------|------|
| 场地预约 | `BK` |
| 课程预约 | `CB` |
| 器材租借 | `ER` |
| 赛事报名 | `TR` |
| 财务流水 | `FN` |
| 穿线服务 | `ST` |
| 财务对账 | `RC` |

### 7.4 逻辑删除约束

- 多数业务表采用 `del_flag`
- 场馆图片、器材图片、部分日志类表更多依赖强关系级联

### 7.5 预约业务约束

当前预约模块的数据库承载规则如下：

1. `SHARED` 仅允许关联 1 块场地
2. `PACKAGE` 允许关联多块场地，但不能覆盖整馆全部场地
3. 同场地同时间段下：
   - 多条 `SHARED` 可并存
   - 存在 `PACKAGE` 时，不允许再创建 `SHARED`
   - 存在任意有效预约时，不允许该场地加入新的 `PACKAGE`
4. 场地状态重算依赖 `biz_booking_court + biz_booking.status`
5. 历史兼容字段 `biz_booking.court_id` 仅保留首场地语义，不再作为冲突检测事实来源

## 8. 已知边界与注意事项

### 8.1 当前事实来源

数据库最终结构以：

```text
src/main/resources/sql/badminton.sql
```

为准。

### 8.2 当前设计边界

- 财务审计与对账表已存在
- 多端业务共享同一套主业务库
- 表结构已能支撑当前 Web、UniApp、教练端业务

### 8.3 仍需持续关注

- 大数据量场景下的分表策略
- 统计分析类查询的读写分离
- 基础资源数据的缓存策略

## 9. 后续优化方向

1. 财务与订单类大表按时间维度做归档或分表评估  
2. 为统计查询准备读写分离与报表库策略  
3. 为场馆、场地、器材等基础数据设计更明确的缓存层
