# 数据库设计文档

> **文档版本**：v3.4.0  
> **项目版本**：1.0.0（功能开发已完成）  
> **最后更新**：2026-05-23  
> **数据库版本**：MySQL 8.x  
> **字符集**：`utf8mb4_unicode_ci`  
> **文档定位**：说明 BMP 数据库范围、表目录、关键关系、业务约束与初始化方式。

## 1. 设计概览

### 1.1 当前事实来源

当前数据库结构以以下脚本为唯一事实来源：

```text
src/main/resources/sql/badminton.sql
```

这个脚本当前已经包含：

- 建库、建表、索引、外键
- 初始化数据
- 预约主单/明细模型
- 三套场地价格字段
- 财务审计、财务对账、用户设置、反馈、取消政策等增量表

### 1.2 当前表规模

基于 2026-05-23 对 `badminton.sql` 的复核，当前库内共有 **27 张物理表**（与 v1.0.0 后端实体一致）。

与旧版文档相比，需要特别纠正两点：

1. 当前库里 **没有** 独立的 `biz_activity` 表，近期动态由接口层聚合生成  
2. 当前库里 **已经有** `biz_cancel_policy`、`biz_feedback`、`sys_user_setting` 等此前未完整同步到文档的表

## 3. 初始化方式

### 3.1 新库初始化

推荐直接执行：

```text
src/main/resources/sql/badminton.sql
```

当前不需要再拼接执行其它拆分 SQL 文件。

### 3.2 初始化后建议检查

建议至少确认以下项目：

```sql
SHOW TABLES;
SELECT id, username, role FROM sys_user WHERE username = 'admin';
```

演示账号（密码均为 **`123456`**，BCrypt 存储）：`admin`（`PRESIDENT`）、`venue_manager_1`（`VENUE_MANAGER`）、`user001`（`USER`）。公开发布仓库前请按需脱敏或删除个人数据。

并确认以下关键表存在：

- `biz_booking`
- `biz_booking_court`
- `biz_finance`
- `biz_finance_audit_log`
- `biz_finance_reconciliation`
- `biz_cancel_policy`
- `biz_feedback`
- `sys_user_setting`

## 4. 关键表说明

字段最终定义以 `badminton.sql` 为准，本章只说明职责和关键语义。

### 4.1 `sys_user`

用途：

- 系统登录用户
- 角色承载主体
- 场馆管理者与教练账号来源

关键字段：

- `username`
- `password`
- `role`
- `venue_id`
- `status`

约束要点：

- `PRESIDENT` 应在应用层保持唯一
- `VENUE_MANAGER` 必须绑定场馆
- `COACH` 账号通过 `sys_coach.user_id` 与教练档案关联

### 4.2 `sys_member`

用途：

- 用户侧消费主体
- 余额、会员等级、充值统计的承载主体

关键字段：

- `user_id`
- `member_name`
- `member_type`
- `member_level`
- `balance`
- `total_recharge`
- `status`

### 4.3 `sys_user_setting`

用途：

- 存储用户级设置项
- 对应当前 `/api/settings/{key}` 的键值配置接口

关键字段：

- `user_id`
- `setting_key`
- `setting_value`
- `create_time`
- `update_time`

### 4.4 `sys_venue` / `sys_court`

用途：

- `sys_venue`：经营资源根节点
- `sys_court`：场馆下属场地资源

`sys_court` 当前已支持三套价格与三项营销开关：

- `package_price_per_hour`
- `shared_price_per_hour`
- `shared_price_per_time`
- `enable_package_hour`
- `enable_shared_hour`
- `enable_shared_time`

### 4.5 `biz_booking`

用途：

- 场地预约主单
- 承载预约头信息与支付状态

关键字段：

- `booking_no`
- `member_id`
- `court_id`（兼容首场地语义）
- `booking_mode`
- `booking_date`
- `start_time`
- `end_time`
- `order_amount`
- `payment_status`
- `status`
- `cancel_policy_id`
- `cancel_fee`
- `refund_amount`

### 4.6 `biz_booking_court`

用途：

- 场地预约明细
- 真实承载一笔预约关联的 1..N 块场地
- 当前冲突检测、占用查询、状态重算的事实来源

关键字段：

- `booking_id`
- `court_id`
- `booking_date`
- `start_time`
- `end_time`
- `pricing_mode`
- `unit_price`
- `duration`
- `line_amount`

当前语义：

- `SHARED`：通常只对应 1 条明细
- `PACKAGE`：允许对应多条明细

### 4.7 `biz_cancel_policy`

用途：

- 配置预约和课程取消规则
- 为取消手续费与退款比例提供策略来源

从初始化数据可确认，当前至少覆盖：

- `BOOKING` 类型取消政策
- `COURSE` 类型取消政策
- 按提前 `24h / 12h / 6h / 0h` 四档划分退款比例

## 6. 当前业务约束摘要

### 6.1 预约约束

当前预约模型的数据库承载规则如下：

1. `SHARED` 仅允许关联 1 块场地
2. `PACKAGE` 允许关联多块场地，但不能覆盖整个场馆全部场地
3. 同场地同时间段下：
   - 多条 `SHARED` 可以并存
   - 存在 `PACKAGE` 时，不允许再创建 `SHARED`
   - 存在任意有效预约时，不允许该场地再加入新的 `PACKAGE`
4. `biz_booking.court_id` 当前只保留首场地兼容语义，不再作为冲突判断事实来源

### 6.2 资源归属约束

以下资源必须归属场馆：

- 场地
- 器材
- 教练
- 赛事
- 多数财务记录

### 6.3 单号前缀约束

| 业务 | 单号前缀 |
|------|------|
| 场地预约 | `BK` |
| 课程预约 | `CB` |
| 器材租借 | `ER` |
| 赛事报名 | `TR` |
| 穿线服务 | `ST` |
| 财务流水 | `FN` |
| 财务对账 | `RC` |

### 6.4 反馈与设置约束

- `biz_feedback` 同时服务联系留言与反馈，不需要拆成两张表
- `sys_user_setting` 采用 `key/value` 结构，适合继续承载轻量个性化配置

## 7. 当前边界与注意事项

### 7.1 当前没有独立活动表

近期动态接口 `/api/activity/recent` 当前直接聚合：

- 最近会员注册
- 最近预约
- 最近财务流水

所以数据库侧不应再引用并不存在的 `biz_activity`。

### 7.2 当前迁移策略偏“全量脚本事实源”

仓库当前更依赖 `badminton.sql` 作为总脚本，而不是维护独立的增量 migration 目录。  
这意味着：

- 新环境初始化更直接
- 老库升级时需要更谨慎地人工对照结构差异

## 8. 后续优化建议

1. 为高频统计与大表查询准备归档 / 分表评估  
2. 为 `biz_finance`、`biz_booking_court` 这类高活跃表继续审视索引策略  
3. 如果后续设置项继续增长，可考虑为 `sys_user_setting` 补充更明确的命名规范与缓存层  
4. 若未来需要更强审计能力，可在财务之外扩展统一业务审计表
