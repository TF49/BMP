# 系统安全漏洞与越权问题修复方案

本文档基于 BMP 当前代码实现进行修正，目标是解决以下几类真实存在的安全问题：

1. 垂直越权：公开注册接口允许调用方自带高权限 `role`，存在自注册为高权限账号的风险。
2. 横向越权：场馆管理员可跨场馆查看不属于本场馆可见范围的会员详情、消费记录，且当前会员列表也缺少场馆级数据隔离。
3. 角色口径不一致：角色字段存在大小写与历史兼容混用，容易导致“鉴权通过但查询/唯一性校验/业务过滤不一致”的问题。
4. 权限异常语义不一致：部分控制器会吞掉权限异常，导致越权场景未能稳定返回预期的拒绝访问结果。

## 现状结论

结合当前代码实现，以下问题确认需要实施修复：

- `AuthController.register` 直接接收并写入前端传入的 `role`，公开注册存在提权面。
- `MemberServiceImpl.findById` 对 `VENUE_MANAGER` 直接放行，场馆管理员可查看任意会员详情。
- `MemberServiceImpl.getConsumeRecords` 对 `VENUE_MANAGER` 未做场馆可见性过滤，场馆管理员可查看任意会员全量消费记录。
- `MemberServiceImpl.findByConditions/countByConditions` 当前对 `VENUE_MANAGER` 仍按全量会员查询处理，存在会员列表遍历风险。
- 多处角色查询与筛选使用精确匹配 SQL（如 `role = #{role}`、`IN ('USER','MEMBER')`），仅依赖运行期 `equalsIgnoreCase` 或 JWT 大写化不足以彻底消除隐患。

说明：

- 当前项目已提供初始化 SQL 内置 `PRESIDENT` 演示账号。
- 当前项目也已提供 `PRESIDENT` 专用后台建号入口 `/api/user/add`。
- 因此，关闭“公开注册高权限角色”不会破坏现有项目初始化与日常运维流程。

## Proposed Changes

### 1. 公开注册提权修复（Authentication & Registration）

#### [MODIFY] [AuthController.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/controller/AuthController.java)

- **修改内容**：
  - 在 `/api/auth/register` 中不再信任前端传入的 `role`。
  - 公开注册仅允许创建普通用户账号，建议直接在服务端强制写为 `USER`。
  - 若仍保留 `role` 字段用于兼容旧前端，则必须先做 `trim + toUpperCase`，仅允许 `USER`，其余一律拒绝。
  - 文案明确提示：`PRESIDENT`、`VENUE_MANAGER`、`COACH` 等角色必须由后台受控分配，不能通过公开注册获得。

#### [MODIFY] [AuthServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/service/impl/AuthServiceImpl.java)

- **修改内容**：
  - 注册逻辑中统一先规范化角色值：`role = role == null ? "USER" : role.trim().toUpperCase()`。
  - 对公开注册来源执行白名单约束，仅允许 `USER`。
  - 保留会长唯一性等防御性校验，但不再允许通过公开注册路径触发高权限建号。

#### [MODIFY] [UserServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/service/impl/UserServiceImpl.java)

- **修改内容**：
  - 将角色规范化下沉到统一写入口，而不是只在公开注册里处理。
  - `register`、`update` 等写用户角色的服务逻辑统一执行 `trim + toUpperCase`。
  - 对空角色继续兜底为 `USER`，避免出现空值、大小写混杂、历史脏数据继续扩散。

#### [MODIFY] [UserController.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/controller/UserController.java)

- **修改内容**：
  - 后台新增用户 `/api/user/add` 与更新用户 `/api/user/update` 也统一做角色规范化，避免后台入口继续写入小写或混合大小写角色。
  - 保证后台是唯一合法的高权限角色分配入口。

### 2. 会员详情与消费记录的数据级隔离（Member Detail & Consume Records Isolation）

#### [MODIFY] [MemberServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/member/service/impl/MemberServiceImpl.java)

- **修改内容**：
  - **修改 `findById` 方法**：
    - 当前 `VENUE_MANAGER` 分支不能再直接无条件返回会员数据。
    - 应复用已有 `isMemberVisibleForVenue(memberId, venueId)` 校验，只允许查看与当前管理场馆存在业务交集的会员。
    - 所谓“业务交集”继续沿用当前判定口径：有该场馆课程预约记录，或有该场馆消费流水。
    - 无可见性时抛出 `AccessDeniedException`。
  - **修改 `getConsumeRecords` 方法**：
    - 当前 `VENUE_MANAGER` 不能再直接走 `findByMemberId(memberId)` / `countByMemberId(memberId)` 全量查询。
    - 应先校验 `isMemberVisibleForVenue(memberId, currentVenueId)`。
    - 通过校验后，仅查询本场馆维度的消费记录：`findByMemberIdAndVenueId` / `countByMemberIdAndVenueId`。
    - 未通过校验时拒绝访问。

### 3. 会员列表的数据范围隔离（Member List Scope Isolation）

#### [MODIFY] [MemberServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/member/service/impl/MemberServiceImpl.java)

- **修改内容**：
  - 当前 `findByConditions` / `countByConditions` 中，`VENUE_MANAGER` 仍与 `PRESIDENT` 共用全量查询口径，此处必须收紧。
  - `PRESIDENT` 可以保留全量会员视图。
  - `VENUE_MANAGER` 应只查询“对当前场馆可见”的会员集合，而不是系统内全部普通用户/会员。
  - 不建议仅在 Service 层拉全量后逐条 `isMemberVisibleForVenue` 过滤，分页、总数与性能都会失真。
  - 建议改为数据库查询阶段直接按场馆交集过滤。

#### [MODIFY] [MemberMapper.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/member/mapper/MemberMapper.java)

- **修改内容**：
  - 新增“按场馆可见范围查询会员列表/统计数量”的 SQL。
  - 建议查询口径为：
    - 仍仅统计 `sys_user.role IN ('USER','MEMBER')` 的会员档案；
    - 再基于课程预约表、消费记录表与当前 `venueId` 建立可见性过滤；
    - 避免 `VENUE_MANAGER` 枚举所有普通会员 ID。
  - 这样可以从根源上修复 `/api/member/list` 横向越权，而不是只修详情接口。

### 4. 权限异常返回语义修正（Forbidden Semantics）

#### [MODIFY] [MemberController.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/member/controller/MemberController.java)

- **修改内容**：
  - 当前部分接口使用 `catch (Exception e)` 直接包装错误信息返回，会吞掉权限异常。
  - 对会员列表、会员详情、消费记录等接口，至少需要对 `AccessDeniedException` 单独 `throw`，交给全局异常处理。
  - 避免越权场景被统一包装成普通失败提示，影响前端区分“无权限”与“系统异常”。

#### [OPTIONAL-RECOMMENDED] [GlobalExceptionHandler.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/common/exception/GlobalExceptionHandler.java)

- **修改内容**：
  - 当前 `AccessDeniedException` 会返回 `Result.forbidden(...)`，但这主要是响应体中的 `code = 403`。
  - 如果希望接口在 HTTP 语义上也严格返回 `403 Forbidden`，建议进一步改为 `ResponseEntity<Result<Object>>` 或显式设置 `@ResponseStatus(HttpStatus.FORBIDDEN)`。
  - 这项不是本次越权修复的唯一前提，但建议同步完善，便于前端和联调工具正确识别权限失败。

---

### 5. 角色大小写与历史兼容统一（Role Normalization）

#### [MODIFY] [AuthController.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/controller/AuthController.java)
#### [MODIFY] [AuthServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/service/impl/AuthServiceImpl.java)
#### [MODIFY] [UserServiceImpl.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/service/impl/UserServiceImpl.java)
#### [MODIFY] [UserController.java](file:///d:/Developer%20Tool/IDEA/JAVAEE/BMP/src/main/java/com/badminton/bmp/modules/system/controller/UserController.java)

- **修改内容**：
  - 所有写入角色的入口统一使用大写标准值：
    - `PRESIDENT`
    - `VENUE_MANAGER`
    - `COACH`
    - `USER`
    - `MEMBER`
  - 所有进入数据库前的角色值统一做：
    - `null` 兜底；
    - `trim()`；
    - `toUpperCase()`；
    - 必要时做合法角色白名单校验。
  - 这样可以避免以下问题：
    - JWT 鉴权时能识别，业务 SQL 却查不到；
    - 唯一性校验使用大写，数据库里却写入了小写变体；
    - `USER/MEMBER` 会员过滤与后台角色查询口径不一致。

#### [OPTIONAL-RECOMMENDED] 历史数据修正

- **修改内容**：
  - 若数据库中已存在小写或混合大小写角色，建议补一次性数据修正脚本，将历史角色统一转成标准大写值。
  - 否则代码修复后，历史脏数据仍可能造成局部业务表现异常。

## Verification Plan

### Manual Verification

1. **公开注册高权限阻断验证**
   - 使用 Postman / Apifox 调用 `/api/auth/register`。
   - 分别传入 `role=VENUE_MANAGER`、`role=PRESIDENT`、`role=COACH`、`role=president`。
   - **预期结果**：
     - 服务端拒绝高权限注册，或直接忽略并强制落为 `USER`；
     - 不得生成高权限账号；
     - 返回信息明确说明高权限角色只能由后台分配。

2. **场馆管理员跨场馆会员详情越权验证**
   - 使用 `venue_manager_1` 登录。
   - 调用 `/api/member/info/{id}` 访问一个仅在场馆 2 有业务交集、与场馆 1 无课程/消费关联的会员。
   - **预期结果**：
     - 后端拒绝访问；
     - 返回权限不足提示；
     - 若同步完成语义修正，HTTP 状态码应为 `403`。

3. **场馆管理员跨场馆消费记录越权验证**
   - 使用 `venue_manager_1` 调用 `/api/member/{id}/consume-records`。
   - 目标会员仅在其他场馆有消费记录。
   - **预期结果**：
     - 查询被拒绝，不能看到其他场馆消费流水。

4. **场馆管理员会员列表遍历验证**
   - 使用 `venue_manager_1` 调用 `/api/member/list`。
   - 对比返回结果中是否仍包含与当前场馆完全无业务交集的普通用户/会员。
   - **预期结果**：
     - 仅返回当前场馆可见的会员集合；
     - 总数与分页结果也应同步收敛，而不是“列表被过滤但 total 仍是全量”。

5. **角色统一规范化验证**
   - 通过后台新增用户或更新用户接口传入小写/混合大小写角色，例如 `user`、`President`、`venue_manager`。
   - **预期结果**：
     - 数据库存储为标准大写角色；
     - 后续用户查询、唯一性校验、会员过滤逻辑表现一致。

6. **历史兼容验证**
   - 使用现有初始化账号 `admin`、`venue_manager_1`、`user001` 登录并验证关键功能。
   - **预期结果**：
     - 修复后不影响现有受控账号体系；
     - 会长后台建号能力保持正常；
     - 普通用户注册仍可正常创建 `USER` 账号。

## Expected Outcome

完成本方案后，应达到以下结果：

- 公开注册无法再获得高权限角色。
- 场馆管理员只能查看与本场馆存在业务交集的会员数据。
- 会员列表、详情、消费记录三类入口的数据范围口径一致。
- 角色写入与查询标准统一，不再出现大小写导致的权限/统计偏差。
- 越权访问在接口层表现为稳定、可识别的权限拒绝结果。
